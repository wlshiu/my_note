[What]Yocto Project --> BitBake 構建引擎

Posted on 2018-09-16 | In linux , make , yocto | 1 | Visitors:

參考書籍:Embedded Linux System with the Yocto Project

  • 書籍比實際的yocto版本要低，所以yocto的詳細命令需要參考其對應版本手冊

運行環境:

  • ubuntu18.04 amd64
  • yocto 2.5.1

BitBake是由純Python開發的構建工具，其靈活性、可擴展性和可移植性遠遠強於同類的 make 這類工具。

其metadata構建規則文件就類似於make中的Makefile。

獲取和安裝BitBake工具

BitBake的代碼位於OpenEmbedded bitbake庫，並且這是個獨立的組件，可以將此工具用於其他工程構建之中。

yocto項目本身就已經包含了BitBake的下載和安裝了，所以無需再來下載。

下載及安裝的方式也是很簡單的: 

#獲取源碼
git clone git://git.openembedded.org/bitbake
#編譯
./setup.py build
#安裝於 /usr/lib/python<version>/site-packages
#也可以不安裝，就在當前目錄使用
./setup.py install

運行BitBake

BitBake的啟動命名為: 

#### 在構建環境的目錄下執行命名 ###
#recipe_name 就是 recipe_name.bb 文件
bitbake <recipe_name / target_name>


#### 在構建環境的目錄外執行命名 ###
#BBPATH需要是絕對路徑!!!
BBPATH="/absolute/path/to/build_env" bitbake <recipe_name / target_name>

配置文件的搜尋

這個過程我感覺和Kconfig的解析流程好像

BitBake啟動後會在當前目錄下搜尋文件 conf/bblayers.conf 以獲得一個基礎構建環境。

  • 其中變量 BBLAYERS 指明了此次構建所包含的層的路徑
  • 而在每個層的 conf/layer.conf 文件給出了當前層的配置文件路徑
    • BBPATH 指定了包含 classes 和 conf 文件夾的路徑，多個路徑通過冒號分隔
    • BBFILES 給出了當前層的具體構建配置文件(recipes files and append files)的路徑。
  • BitBake會用變量 LAYERDIR 來保存層的頂級路徑，BitBake便順藤摸瓜找到各個層的配置文件。
  • BitBake在遍歷過程中，會根據 BBFILE_COLLECTIONS, BBFILE_PATTERN_<name> , BBFILE_PRIORITY_<name> 來整理配置
    • BBFILE_COLLECTIONS 指定一個蒐集器，將相同的文件聚集到一起
    • BBFILE_PATTERN_<name> 以正則表達式的方式指定BitBake的起始搜尋路徑
    • BBFILE_PRIORITY_<name> 指定此層配置文件的優先級(1~10,數值越高優先級越高)
      • 當有同名的recipe文件，BitBake使用優先級高的
      • 當有多個append文件，BitBake先應用優先級高的append文件，再應用優先級低的
  • BitBake在解析完 bblayers.conf 文件後，還會去解析 conf/bitbake.conf 文件，以獲取編譯環境的設置。

BitBake命令行

BitBake的命令參數很多，下面對它們進行分類整理。

以包含依賴的方式處理構建

BitBake默認的構建任務是由變量 BB_DEFAULT_TASK 所指定的 build 任務。

build 任何在正式啟動編譯之前會評估好各個軟件包之間的依賴和編譯順序。

一般情況下使用的 bitbake <recipe_name / target_name> 命令就是啟動的 build 任務。

  • 當出現了錯誤時，bitbake將停止執行任務，可以使用 -k 選項讓其繼續執行。

當編譯軟件包時，默認會編譯其最新版本或者是由變量 PREFERRED_VERSION 指定的版本。 可以在目標名稱後加上版本名以指定編譯特定版本: 

bitbake editor-1.0
bitbake editor-2.0-r3

以不包含依賴的方式處理構建

可以指定只編譯某個recipe文件的規則，而不進行依賴編譯。

  • 當編譯出錯時則退出
bitbake -b <recipe_name>.bb
bitbake --buildfile=<recipe_name>.bb

執行指定的任務

有以下兩種方式:

  • 包含依賴的方式執行
bitbake <packet> -c <task_name>
bitbake <packet> --cmd=<task_name>
  • 不包含依賴的方式執行
bitbake -b <recipe_name>.bb -c <task>
bitbake -b <recipe_name>.bb --cmd=<task>

強制執行

BitBake會為每個完成的任務打上時間戳，以完成增量編譯，用戶可以取消此時間戳的判斷以強制編譯該任務 

bitbake <packet> -C <task>
bitbake <packet> --clear-stamp=<task>
bitbake <packet> -c <task> -f
bitbake <packet> -c <task> --force

BitBake將任務的輸出放在 "shared state cache"中(也就是 sstate-cache 文件夾)，當任務輸入並沒有被改變時， BitBake直接從 sstate-cache 中取出結果，而不需要重新編譯。

  • 這也是為什麼建議將 sstate-cache 放在 build 之外的目的，多個 build 可以共享編譯輸出，減小編譯時間

用戶可以關閉 sstate-cache 的使用: 

bitbake <packet> -c <task> --no-setscene

顯示metadata

當用於調試構建過程時，需要輸出BitBake所分析到的元數據文件、變量、功能等等: 

#顯示所有信息
bitbake -e
bitbake --environment
#僅顯示關於此 <package>的信息
bitbake -e <package>
bitbake --environment <package>
#顯示recipes的版本
bitbake -s
bitbake --show-versions

創建依賴圖

BitBake基於graphviz表示包之間的依賴圖: 

#生成描述依賴的dot文件
bitbake -g <package>
bitbake --graphviz <package>

#生成描述依賴的dot文件，並忽略某個包
bitbake -g <package> -I <ignore_package>

最終會在構建目錄下生成下面3種DOT語言描述文件:

  • task-depends.dot : 描述任務級之間的依賴
  • recipe-depends.dot : 包與包之間的依賴

然後使用 graphviz 生成圖片即可，例如: 

dot -Tpng -o pn-depends.png pn-depends.dot

注意：這張圖很大，建議使用 tred去掉中間依賴 

tred recipe-depends.dot > recipe-depends-notrans.dot

也可以使用BitBake所自帶的依賴描述工具: 

bitbake -g -u taskexp <package>

配置的設置和覆蓋

用戶可以在BitBake解析其他文件前後加入附加的配置文件:

  • 在解析其他文件前加入配置文件，一般用於設置環境變量
bitbake -r <prefile>.conf <target>
bitbake --read <prefile>.conf <target>
  • 在解析其他文件後加入配置文件，一般用於覆蓋前面配置文件的設置
bitbake -R <postfile>.conf <target>
bitbake --postread <postfile>.conf <target>

服務器

BitBake是一個典型的CS架構下的應用程序，每次啟動BitBake時它都會啟動一個服務端程序(叫做 cooker )， 和一個客戶端程序。

服務器會啟動多個線程分析元文件，並進行編譯。服務器和客戶端使用pipe來交互信息。

當然，這種架構完全可以用於遠程編譯：小組裡幾個人共用一個強大的服務器，將自己客戶端的代碼放在服務器編譯後取得結果。

  • 在服務器上啟動BitBake服務端程序:
bitbake --server-only --bind=<ip>:<port>
  • 客戶端連接 BitBake服務器構建:
bitbake --remote-server=<ip>:<port> <target>

BitBake元數據

BitBake通過分析元數據來控制構建過程，元數據來描述軟件包該如何構建以及它的依賴關係。

BitBake將分析後的元文件放在一個緩存文件夾中，避免下次重複分析而浪費不需要的時間。

BitBake分析以下兩類元數據:

  • 變量(Variables)： 包括控制整體的全局變量，以及控制一小部分的局部變量
  • 可執行元數據(Executable Metadata)：在recipes和classes中嵌入的功能和命令

元數據文件由以下5種文件組成：

  • 配置文件(.conf):配置文件中的變量都是全局變量，所以它們會影響全局。
    • 如果多個配置文件中有多個相同的變量，那麼使用優先級最高的那個。
      • bitbake.conf 優先級最低，而構建目錄下的 local.conf 優先級最高
  • recipe文件(.bb):recipe文件用於描述軟件包以及該如何構建此軟件包
    • 對包的操作命令通常有:下載、解壓縮、打補丁、編譯、打包、安裝
  • class文件(.bbclass):類文件主要包含很多recipe文件所共有的設置，以便它們繼承。
    • BitBake會搜尋當前層下的 classes 子文件以獲取類文件
    • recipe中使用 inherit 指令來繼承類文件
      • 類文件是全局的，所以recipe可以在任何位置繼承
  • append 文件(.bbappend) : 附加文件是對recipe文件的擴展，附加文件的前綴名稱必須與recipe文件一致
    • append文件中包含的層路徑也要與recipe文件一致
    • 當不同層中有一樣的append文件時，由層的優先級來決定使用哪一個
  • include 文件(.inc): 每中元數據文件都可以使用 include, require 指令包含其他文件,include文件表示其他元文件所共同包含的元文件
    • 包含執行指令的元文件只能被 recipes,append,classes文件包含

元數據的語法

元數據的語法類似 Makefile和shell 腳本。

註釋

使用 # 作為註釋開頭， 但註釋分為普通註釋和功能函數註釋

  • 普通註釋(在函數之外)的 # 必須在 行的第一列 才符合語法規則
  • 元文件中可以包含 shell 和 python 功能函數，在函數內部的註釋就比較隨意了。

變量

元數據中的變量對於BitBake來說都是字符串，並且其和Makefile一樣，引用變量使用 ${} 實現。

變量名

變量名中可以包含:大小寫字母、數字、下劃線(_)、橫線(-)、點(.)、加號(+)、波浪線(~)

為了與Makefile和shell保持一致的習慣，一般還是使用大寫字母和下劃線的形式來組成變量名。

變量的作用域

  • 在配置文件(.conf)中定義的變量都是全局變量，所有的recipes文件可見
  • 在recipes文件中定義的變量是局部變量，當其修改全局變量時，其作用也是局部的

變量的賦值

變量的賦值是使用單引號或雙引號包含的字符串(一般都使用雙引號)，具有以下幾種賦值類型:

  • 直接賦值( = ) : 在使用時才真正賦值
#在雙引號字符串中的其他引號字符可以使用轉義字符(\)或單引號來表示
VAR1 = "This is a \"quotation\" inside a variable assignment"
VAR2 = "This is a second 'quotation' inside a variable assignment"

VAR3 = "jumps over"
VAR4 = "${VAR3} the lazy dog."
VAR3 = "falls on"
#在真正使用VAR4的時候，其值為 "falls on the lazy dog."
  • 默認賦值( ?= ): 當變量為空時，為其指定一個默認值
#如果A之前沒有被賦值，那麼A的值就是 value1
A ?= "value1"
#如果B之前沒有被賦值，那麼B的值就是 value2
B ?= "value2"
B ?= "value3"
#無論C之前有沒有被賦值，其值都是 value5
C ?= "value4"
C = "value5"
  • 弱賦值 ( ??= ) : 如果變量之前沒有被賦值，那麼其值在分析時被確定，其優先級 < 默認賦值 < 直接賦值
#如果A之前沒有被默認賦值和直接賦值，那麼其值就是 value1
A ??= "value1"
#如果B之前沒有被默認賦值和直接賦值，那麼其值就是 value3
B ??= "value2"
B ??= "value3"
#如果C之前沒有被默認賦值和直接賦值，那麼其值就是 value4
C ?= "value4"
C ??= "value5"
#D的值就是 value6
D = "value6"
D ??= "value7"
  • 立即賦值( := ) : 立即為變量設置值
VAR1 = "jump over"
VAR2 := "${VAR1} the lazy dog"
VAR1 = "falls on"

#在使用VAR2時其值為 "jump over the lazy dog"


#BitBake可以解析python語法，在表達式前面加 @即可
DATE = "${@time.strftime('%A %B %d, %Y', time.gettime())}"
TODAY := "Today is: ${DATE}."
  • 值的追加:
    • "+=" : 在現有值的後面增加值，並自動添加空格
    • "=+" : 在現有值的前面增加值，並自動添加空格
    • ".=" : 在現有值的後面增加值，不會添加空格
    • "=." : 在現有值的前面增加值，不會添加空格
    • "_append" : 在名稱後增加 "_append" 表示在值後面添加值，不會添加空格
    • "_prepend" : 在名稱後增加 "_prepend" 表示在值前面添加值，不會添加空格
  • 值的去除：在名稱後加 "_remove" ，代表在原值中去掉以空格做分隔的哪部分
VAR1 = "123 456 789 123456789 789 456 123 123 456"
VAR1_remove = "123"
VAR1_remove = "456"

#最終VAR1的值為 "789 123456789"
  • 條件賦值：可以為變量的賦值設定一定的條件，當滿足該條件值才可以被改變。這在構建過程常用
#OVERRIDES變量中包含了賦值的條件，條件使用冒號分隔，條件優先級從左到右依次升高
#也就是說最後邊的優先級是最高的,其中默認的 "append" 和 "prepend" 具有最高優先級
OVERRIDES = "sun:rain:snow"

#變量使用後綴來確定使用哪個條件
#最終變量 PROTECTION中的值為 "sweater" 因為snow的優先級高於 "rain"
PROTECTION = "unknown"
PROTECTION_snow = "sweater"
PROTECTION_rain = "umbrella"

#也可以附加值
PROTECTION_append_rain = "umbrella"

包含

metadata 文件可以包含其他文件可以獲取它們的設置，BitBake使用以下兩種命令包含其他文件:

  • include : 可選包含，當BitBake沒有找到該文件時會繼續執行不會報錯
  • required : 必須包含，當BitBake沒有找到該文件時會報錯並停止執行

recipes 和 classes 文件可以包含配置設置和命令設置文件，而配置文件只能包含僅有配置設置的文件！

.inc 文件可以包含其他文件，當包含形成一個環形包含時，BitBake會給出報錯信息

當有多個同名文件時，BitBake以找到的第一個文件為準。 

#兩個命令都可以使用絕對路徑和相對路徑
#當使用相對路徑時，它是相對於變量 BBPATH 的路徑
include file1.inc
include meta-test/recipes-core/images/file2.inc
required /home/build/yocto/file3.inc

和預編譯處理過程一樣，包含的文件會在包含指令的位置處展開。

繼承

recipes,append,classes文件可以繼承classes文件，繼承使用命令 inherit <class_name>

BitBake繼承class文件的流程為:

  1. 通過 inherit 命令中的 <class_name> 找到 <class_name>.bbclass 文件。
    • <class_name>.bbclass 文件位於 classes 文件夾中， class_name 需要是獨一無二的
  2. 在解析完配置文件之後以及recipes文件之前，BitBake開始分析classes文件
  3. 在分析classes文件中遇到了文件包含，則順序展開即可
  4. BitBake使用寫時複製(copy on write, COW)，來保持class副本

可執行語句

可執行語句可以存在於 recipes,append,classes 文件中，可以是 shell 或 python 功能函數。

BitBake將功能函數名作為變量名，將其函數體作為右值，所以這些功能函數也可以像其他普通變量一樣被擴展。

在classes中的功能函數為全局作用域，而在recipes和append中的功能函數為局部作用域。

python本地函數

python本地函數的定義有些許不同，需要使用 python 關鍵字來提示BitBake使用python解釋器來執行:

  • 在函數內的就可以按照標準的python語法來實現了
python printdate () {
    import time
    print time strftime(『%Y%m%d', time.gettime())

python全局函數

當使用標準語法來定義函數時，此函數就是一個全局的函數，可以被其他的metadata所使用

python匿名函數

匿名函數在當前文件被解析完後自動執行，有以下兩種定義方式:

# Anonymous function using the __anonymous keyword
...
}

# Anonymous function omitting the function name
...
}

任務(tasks)

上面所講述的幾種函數，將他們名稱前加上 do_ 就被稱為任務(tasks)，BitBake使用 addtask 命令來將此任務加入任務列表。 

python do_clean () {
# task to clean remove build output...
}
addtask clean
do_build () {
# task to build the software package
...
}
addtask build
do_download () {
# task to download sources
...
}
addtask download before do_build
do_unpack () {
# task to unpack sources
...
}
addtask unpack before do_build after do_download
do_compile () {
# task to compile software package
...
}
addtask compile before do_build after do_unpack
do_install () {
# task to install binaries, libraries, and documentation
...
}
addtask install after do_build
  • 上述定義了任務 do_clean() ,然後使用 addtask clean 將其加入任務列表，BitBake可以使用命令行的方式來指定執行此任務(比如recipe文件的名稱為 myrecipe.bb)
    • bitbake myrecipe -c clean
  • 上面的 addtask download before do_build , addtask unpack beofre do_build after do_download , addtask compile before do_build after do_unpack , addtask install after do_build
    • 當執行 bitbake myrecipe -c build 時，其執行函數順序依次為 do_download -> do_unpack -> do_compile -> do_build -> do_install
      • 也就是說將這些任務串聯起來了
  • 當recipe中調用 BitBake時沒有為其指定任務時，則其使用 BB_DEFAULT_TASK 所定義的任務，默認為 build
BB_DEFAULT_TASK ?= "build"

在函數中訪問BitBake變量

  • 在shell函數中訪問：直接使用變量引用訪問即可
    • shell中改變了變量，其作用域在函數內
  • 在python函數中訪問：需要使用方法才可以
    • 使用全局變量中的 d 方法 getVar() 和 setVar 來獲取和設置此值
      • getVar() 參數2用於表示是否要使用此變量被擴展後的值(當變量值中包含其他變量時，是使用其他變量的值還是按照原字符串返回)
    • 注意： 使用這種方式修改的變量則是全局修改的，為了局部修改可以使用 createCopy() 方法來創建變量副本，也是使用COW的方式
      • localdata = d.createCopy()

源碼的下載

鏈接獲取及其格式

在進行正式構建前的第一步就是源碼的獲取，BitBake是從上游庫(Upstream repositories)獲取源碼。

上游庫可以是本地路徑、遠程文件服務器、git庫等等，而BitBake會根據庫的路徑來自動使用合適的協議獲取源碼。

同時源碼也可能以各種不同的格式被壓縮,而BitBake也會根據源碼的壓縮格式來自動使用合適的工具解壓縮。

BitBake提供 fetch2 庫(位於 bitbake/lib/bb/fetch2),使用Python其 Fetch 類來實現源碼的智能獲取。

描述鏈接的格式為: 

<scheme>://[<user>[:<pwd>]@]<host>[:<port>]/<path>[;<key>=<value>;..]
  • 對於本地路徑，支持絕對路徑和相對路徑兩種方式:
file:///absolute/path/to/file
file://relative/path/to/file

使用 Fetch 類

為了可以獲取源碼，需要使用 Fetch 類來觸發源碼的下載:

  • 最終的下載會指向 DL_DIR 所指定的目錄

實際上OE Core的 base.bbclass已經提供了此任務,無需用戶實現

源碼獲取的具體實現

本地源碼的獲取

本地源碼的獲取支持絕對路徑和相對路徑兩種格式:

  • 本地源碼的下載其實就是將路徑中的文件複製到 DL_DIR 指定的路徑中
  • 本地文件獲取的執行源碼位於 bitbake/lib/bb/fetch2/local.py
SRC_URI = "file:///absolute/path/to/file"
SRC_URI = "file://relative/path/to/file"

相對路徑使用 FILESPATH 和 FILESDIR 來定位文件的位置:

  • FILESPATH : 此變量使用冒號分隔多個路徑，提取器(fetcher)依次搜尋這些目錄直到找到匹配的文件
    • 當有多個路徑下有同名文件時，提取器下載第一個找到的文件
  • FILESDIR : 當 FILESPATH 中沒有路徑時，提取器會檢查變量 FILESDIR 中的路徑

如果沒有找到對應的文件，提取器就會報錯。

HTTP/HTTPS/FTP 源碼的獲取

BitBake默認使用wget來完成http,https,ftp協議的文件下載。

  • 源碼腳本位於 bitbake/lib/bb/fetch2/wget.py

可以通過設置變量 FETCHCMD_wget 來設置 wget 的選項，可設置的值有:

  • md5sum : 使用md5校驗，如果與提供的校驗碼不匹配則報錯
  • sha256sum : 使用sha256sum校驗，如果與提供的校驗碼不匹配則報錯
  • downloadfilename : 將下載好的文件重命名為當前設定的文件名
  • name : 鏈接地址的別名

其中 md5sum 和 sha256sum 至少需要提供其中一個！ 

SRC_URI	=	"http://host.site.org/downloads/srcpkg.tar.gz;md5sum=12345"
SRC_URI	=	"https://host.site.org/downloads/srcpkg.tar.gz;sha256sum=6789"
SRC_URI	=	"ftp://user:pwd@host.site.org/repo/srcpkg.tar.gz;md5sum=12345"
SRC_URI	=	"ftp://host.site.org/srcpkg.tar.tgz;md5sum=12345;downloadfilename=file.tgz"

校驗碼一般都比較長，所以可以分離URL和校驗碼: 

#當只有一個下載包時
SRC_URI	=	"http://host.site.org/downloads/srcpkg.tar.gz"
SRC_URI[md5sum]	=	"12345"
SRC_URI[sha256sum]	=	"6789"

#當有多個下載包時，通過名稱來區分
SRC_URI	=	"http://host.site.org/downloads/srcpkg1.tar.gz;name=pkg1
           ftp://host.anothersite.org/downloads/srcpkg2.tgz;name=pkg2"
SRC_URI[pkg1.md5sum]	=	"12345"
SRC_URI[pkg2.sha256sum]	=	"6789"

SFTP 源碼的獲取

在使用SFTP前，用戶需要先用ssh key提前獲取權限後，再來讓bitbake以sftp的方式抓取，不能直接在URI中輸入密碼。

  • 源碼腳本位於 bitbake/lib/bb/fetch2/sftp.py
SRC_URI	=	「sftp://host.site.org/downloads/srcpkg.tgz;md5sum=12345」
SRC_URI	=	「sftp://user@host.site.org/downloads/srcpkg.tgz;md5sum=12345」

git 源碼的獲取

git 抓取腳本位於 bitbake/lib/bb/fetch2/git.py

抓取器提供了以下參數可以控制clone的過程:

  • protocol : 支持的協議有 git,file,ssh,http,https,rsync
    • 默認的便是 git協議
  • branch : 要切換到哪個分支
    • 默認為 master
  • name : 分支的名稱
  • tag : 切換到哪個標記
    • 默認為 HEAD
  • rebaseable ： 使用 "rebaseable=1" 告知上游庫可能以後會有 rebase 操作，抓取器需要將當前工作目錄緩存
    • 默認為 0
  • nocheckout : 使用 "nocheckout=1" 告知不要切換到抓取後的分支
    • 默認為 0
  • bareclone : 使用 "bareclone=1" 告知僅clone裸庫而不釋出源碼
    • 默認為0
#注意：最開始的地址都是以默認的git協議寫的，具體的protocol需要在後面更改
SRC_URI	=	"git://git.site.org/git/repo.git;branch=develop"
SRC_URI	=	"git://git.site/org/git/repo.git;tag=0C12ABD"
SRC_URI	=	"git://git.site.org/git/repo.git;protocol=http"

當庫代碼中有子模塊時，提取器會在clone完基礎代碼後再clone子模塊。 git 抓取腳本位於 bitbake/lib/bb/fetch2/gitsm.py

源碼鏡像

BitBake支持從鏡像站點下載軟件包，其訪問順序為:

  1. 根據 SRC_URI 中提供的軟件包，在 DL_DIR 指向的路徑中尋找是否有匹配的文件。如果存在此文件，則無需從遠程下載了。
    • 當軟件包使用了SCM倉庫，那麼BitBake會檢查 DL_DIR 中的分支和tag是否對應。
  2. 當 DL_DIR 中沒有對應的軟件包時，BitBake根據 PREMIRRORS 中提供的鏡像站點下載文件
    • 如果不想使用此鏡像站，那麼將變量 PREMIRRORS 的值設置空字符串即可
  3. 如果在 PREMIRRORS 的鏡像站點沒有找到對應的包，BitBake則會在 SRC_URI 指向的路徑下載軟件包
  4. 如果在 SRC_URI 路徑下載失敗，那麼BitBake會在 MIRRORS 指向的備用鏡像站點下載文件
    • 如果不想使用此鏡像站，那麼將變量 MIRRORS 的值設置空字符串即可
  5. 備用鏡像下載文件都失敗的話，BitBake則會報錯。

PREMIRRORS 和 MIRRORS 使用正則表達式來列出各種下載協議對應的站點: 

MIRRORS	=	"\
ftp://.*/.*				http://downloads.mirrorsite.org/mirror/sources/	\n	\
http://.*/.*			http://downloads.mirrorsite.org/mirror/sources/	\n	\
https://.*/.*		http://downloads.mirrorsite.org/mirror/sources/	\n	\
git://.*/.*				http://downloads.mirrorsite.org/mirror/sources/	\n	\
svn://.*/.*				http://downloads.mirrorsite.org/mirror/sources/	\n"

BitBake會根據 SRC_URI 對應的協議來找到鏡像站點的鏈接並做替換。

yocto為 poky也提供了默認的鏡像站。

創建自己的鏡像站

可以事先從yocto中下載好軟件包，然後自己搭建一個內部局域網的ftp服務器，這樣就可以小組內共享了。
