GNU GDB調試手冊

開發基礎
  • 內核
  • 後台開發
  • 開發基礎
  • 讀書筆記

　　人家都說Windows平台下的程序員是幸福的，因為Visual Studio實在是太好用了。
　　我想一聽到上面這句話，絕大多數Linux平台C/C++程序員都將會沉默，至少網上搜Linux平台下C/C++開發調試，很大一部分文章都是介紹搭建IDE的，這也側面說明了Linux平台下沒有一個佔絕對主流地位好用的IDE。自己平時寫程序是用的SlickEdit，這個IDE很貴，支持代碼補全和跳轉，調試過程也支持斷點、Watch等特性，但是使用過程中還是有這樣那樣的小問題(比如非英文字串顯式、速度比較慢)，但是總體比較而言還是比較優秀的跨平台C/C++ IDE。雖然線下編碼調試使用這貨當然可以，然則線上環境就無能為力了，再加上自己之前對gdb也只是瞭解個表明，這次就順著gdb的文檔深挖一下！
　　當然，對於那種持續添加跟蹤代碼，重新編譯程序，然後運行程序並分析跟蹤代碼輸出，修正程序錯誤之後再刪除跟蹤代碼，並且針對發現的每個新的程序問題都重複上面步驟的同學，即使某些時候確實能達到調試跟蹤的效果，但遲早也會被自己累死(蠢死)。
　　真是一看嚇一跳，gdb的命令行調試要遠比IDE的功能高級的多，如果只是通常的設置斷點，監測變量什麼的可能IDE比較方便，但是一旦上升到使用那些高級點的調試技巧，反而gdb在命令行的模式下更為的方便和高效，起碼啟動速度和響應速度會高很多。
　　前方預警：這將會是一篇很長很長的文檔摘讀，而且個人翻譯的可能比較生硬，期待改進。

一、開始使用gdb

1.1 啟動gdb

　　gdb接需要調試的程序可執行文件，後面可以附加core文件或者pid進程號(調試正在運行的進程)，如果有第三個參數gdb總是先嘗試解釋他為core文件，然後才是進程號。第二種格式採用–args，待調試的程序執行的時候需要額外的參數，此時可執行程序後面的參數不再被gdb解釋。
　　組合鍵Ctrl-D或者輸入quit可以退出調試狀態，如果之前attach到一個進程上面去了，可以用detach命令釋放進程，如果quit有表達式那麼表達式就作為gdb的退出狀態，否則是被調試的子進程的退出狀態。gdb中使用Ctrl-C不會終止gdb，而是會終止gdb正在運行的調試指令返回到gdb命令狀態，這條特性有時候也十分重要，比如程序陷入了死循環，此時通過Ctrl-C暫停程序的執行，然後通過backtrace就可以查看整個死循環的調用棧，從而幫助定位問題。
　　在gdb中可以使用shell或者!做前導來執行shell中的命令而不用退出gdb(gdb的退出和重啟動的代價是很大的)，make命令可以用來重新編譯程序而不用先導shell|!符號，而gdb自動重新加載符號。
　　通過set logging on可以打開日誌功能，默認會將日誌輸出追加到當前目錄的gdb.txt文件中。

1.2 gdb中的命令

　　gdb是很智能的，所有的命令都可以使用TAB、TAB-TAB進行補齊或者提示，不輸入任何命令的空換行(在可能的情況下)表示重新執行前一條指令，這節省了重複輸入step|list等指令的勞動。gdb允許指令和參數的縮減輸入，沒有歧義的時候會運行，而在遇到歧義的時候起會提示出各個歧義的命令列表供參考。
　　有些命令參數中可能含有特殊的符號(比如括號)，尤其在C++允許函數重載的時候需要使用參數類型來進行區分，此時參數需要使用 『 單引號來包圍，而且在需要的時候gdb也會自動幫你包圍並給出候選列表的。

1.3 gdb幫助

　　gdb本身附帶了很多幫助命令，比如：
　　help command：顯示名的幫助信息
　　apropos args：會在命令名、文檔中搜索指定的表達式
　　info args|registers|breakpoints|…：可以顯示很多狀態信息，具體可以查看help info
　　set var val：設置環境變量
　　show var：顯示環境變量的值

1.4 調試執行程序

　　對於待調試程序編譯時候，總應該使用-g選項生成調試信息。

1.4.1 啟動調試程序

　　run/r
　　必須在gdb啟動的時候指定調試文件，或者啟動後再使用exec|exec-file指定調試文件。啟動的時候gdb會創建子進程(inferior process)並在子進程中立即運行該程序。run命令後面可以接參數，所有的參數都會被傳遞給可執行程序，而下次調試如果直接run而不添加參數，上一輪的參數會被繼承自動添加(下面的start也是一樣)。同時，GDB會自動保存之前設置的斷點等信息，這也是推薦啟動GDB不要輕易退出，即便是重新編譯程序的情況下，通過run可以讓目標程序重新運行，而之前設置的斷點等信息不會丟失。
　　除了在調用run/start的時候指定參數，還可以單獨調用set args方法設置參數。參數的展開和shell規則一樣，而且支持輸入輸出重定向(比如>aa.txt)，展開後傳遞給run/start使用。
　　start
　　該命令會在程序入口函數的地方(C/C++默認是main)添加一個臨時斷點，然後程序立即運行並在入口函數的地方被暫停。
　　此處還需要注意，C++程序在執行main之前會創建static和全局變量，他們的構造函數會被調用，所以程序入口函數不一定會被最早執行的程序代碼。

1.4.2 調試運行的進程

　　attach process-id
　　可以用來跟蹤一個已經在運行的進程，然後被跟蹤的進程會暫停執行，然後可以做任何的調試操作，以及continue讓程序繼續執行。同時，gdb還會在當前工作路徑、源代碼搜索路徑等地方搜索該進程的可執行文件，當然也可以使用file命令手動加載可執行文件。
　　detach
　　一旦detach之後程序會繼續執行，而且和gdb再無瓜葛了。如果當前有attach的進程而使用Ctrl-D退出gdb，gdb會提示是否要detach調試的進程。
　　kill
　　該命令會kill掉當前調試的進程，無論是run/start創建的還是attach調試的。常用情況是修改代碼後需要重新調試，此時可以先kill掉當前調試，然後編譯出新的可執行程序，再使用run/start開啟調試，此時gdb會發現可執行程序已經修改了，會重新讀取符號表信息(但是之前的斷點等信息還是得以保留的)。

1.4.4 調試多線程程序

　　線程除了共享地址空間之外，擁有自己獨立的寄存器狀態、執行棧和線程存儲。gdb可以查看跟蹤進程的所有線程的狀態信息，但是只能有一個線程處於當前被跟蹤的狀態。
　　info threads [thread-id-list]
　　顯示所有(或者指定)線程的信息，帶星號標識的是當前跟蹤的線程。
　　thread thread-id
　　切換thread-id線程為當前跟蹤線程。
　　thread apply [thread-id-list | all] command
　　在所有(或者指定)線程線程上執行command命令。
　　thread name [name]
　　設置線程的名字(info threads中會顯示)，如果不帶參數，會恢復名字為操作系統默認名字。

1.4.5 調試fork創建的多進程程序

　　GDB對fork產生新進程的調試沒有特殊的支持，當使用fork產生子進程的時候，GDB會繼續調試父進程，子進程不會被阻礙而是繼續執行，而此時如果子進程遇到斷點後，子進程會收到SIGTRAP信號，當在非調試模式下默認會導致子進程中止。如果要調試子進程，可以讓子進程啟動後睡眠一段時間(或者監測某個文件是否創建)，在這個空隙中使用ps查看子進程的進程號，然後再啟動gdb去attach到這個進程上面去調試。
　　通過catch命令可以讓gdb在fork/vfork/exec調用的時候暫停執行。
　　影響fork多進程調試還有以下幾個變量：
　　follow-fork-mode
　　可選值是parent或child，指明當正在被調試的進程fork出子進程的時候，如果該值是parent，那麼父進程被調試子進程正常執行，這也是默認行為；而當其為child的時候，則新創建的子進程被調試，父進程正常執行。
　　detach-on-fork
　　可選值是on或off，知名當被調試的進程fork出子進程後，是否detach某個進程還是調試他們倆，如果是on，那麼子進程(或父進程，取決於上面follow-fork-mode的值)將會被detach正常執行，這是默認行為；否則兩個進程都會被gdb控制，其中的一個進程被正常調試，另外一個進程被suspend住。
　　follow-exec-mode
　　gdb對於exec調用的反應，其值可以為new或same，當值為new的時候，之前fork的子進程和exec後的進程都會被保留；如果是same，則exec的進程使用之前fork的進程，exec會用新的執行鏡像替代原先的可執行程序，這是默認行為。

1.4.6 設置書籤保存歷史狀態

　　gdb可以在程序執行的過程中保留快照(狀態)信息，稱之為checkpoint，可以在進來返回到該處再次查看當時的信息，比如內存、寄存器以及部分系統狀態。通過設置checkpoint，萬一調試的過程中錯誤發生了但是已經跳過了錯誤發生的地方，就可以快速返回checkpoint再開始調試，而不用重啟程序重新來過。
　　checkpoint
　　對當前執行狀態保存一個快照，gdb會自動產生一個對應的編號供後續標識使用。從提示信息看來，其實每建立一個checkpoint，都會fork出一個子進程，所以每個checkpoint都有一個唯一的進程ID所對應著。
　　info checkpoints
　　查看所有的checkpoint，包括進程ID、代碼地址和當時的行號或者符號。
　　restart checkpoint-id
　　恢復程序狀態到checkpoint-id指明的checkpoint，所有的程序變量、寄存器、棧都會被還原到那個狀態，同時gdb還會將時鐘信息回退到那個點。恢復過程中程序的狀態和系統的部分狀態是支持的，比如文件指針等信息，但是寫入文件的數據、傳輸到外部設備的數據都無法被回退。
　　delete checkpoint checkpoint-id
　　刪除指定的checkpoint。
　　此外，checkpoint的作用還在於斷點、觀測點不是什麼情況下都可用的情況下，因為Linux系統有時候為了安全考慮，會隨機化新進程的地址空間，這樣重啟調試程序會導致之前使用絕對地址設置的斷點、觀測點不可用。

二、中斷和繼續運行

2.1 斷點、監視點和捕獲點

　　gdb除了普通斷點，還支持檢測點(watchpoint)、捕獲點(catchpoint)兩類特殊的斷點類型，他們都能使用enable、disable、delete進行管理。檢測點會在檢測的變量以及使用運算符組合成的表達式的值發生改變的時候暫停程序的執行；catchpoint是在某個事件發生的時候暫停程序的執行，比如C++拋出異常、加載庫的時候。
　　還需注意的是，GDB顯示的是將要執行的代碼行，而非執行後的代碼行！

2.1.1 普通斷點

　　break [location]
　　location可以是函數名、行號、指令地址，在程序執行到這些指定位置前會暫停下來(暫停時候這些位置的任何代碼都不會被執行)。如果沒有location參數，則斷點會在下一條執行的指令前設置，在最內幀中gdb會在下次執行到該點的時候暫停，這種情況最常見的是在循環中使用。
　　使用break +/-offset，可以在當前選中幀中正在執行的源代碼行的前面或者後面設置偏移位置的斷點。
　　break … if cond
　　每次指定到該點的時候會先計算cond的狀態，當cond!=0的時候斷點生效，程序執行暫停下來。而且在同一個位置，可以使用不同的條件設置多個條件斷點。
　　tbreak args
　　只生效一次的臨時斷點，一旦斷點生效程序暫定在這個斷點處的時候，該斷點會被自動刪除掉。
　　rbreak regex
　　在所有匹配regex正則表達式的函數名處都設置斷點，並打印出這些新設置的斷點。正則表達式的元字符含義同grep工具的含義(而同shell有所差別)。這在C++設置重載函數斷點的時候十分有效。
　　rbreak file:regex
　　通過一個文件名的前綴，可以縮小rbreak設置斷點的範圍(否則庫的頭文件也會被搜索並設置斷點)。
　　info break|breakpoints
　　查看斷點(同時還包括watchpoint和catchpoint)，同時如果斷點被觸發了，還會顯示觸發的次數信息。

　　在一個位置上設置斷點，其本質上可以對應於多個位置，比如：同名重載函數、C++合成的多個構造函數、模板函數和模板類、inline函數。對於這種單個斷點多個地址的情況，gdb會自動在需要的位置插入斷點，當使用info break顯示的時候，會以breakpoint-number.location-number的方式顯示，斷點管理可以引用breakpoint-number，或者針對某個子地址的斷點操作。
　　動態庫也支持斷點，不過其地址需要在加載庫的時候才能解析，因此動態庫中的斷點在未能解析地址前都是pend狀態，同時動態庫在程序執行的過程中也可能進行多次加載和卸載的操作，gdb在只要有動態庫加載和卸載操作的時候，都會重新計算斷點的位置信息。

2.1.2 監視點

　　監視點是監視特定內存位置、特定表達式的值是否改變而觸發程序暫停執行，而不用去關心該值到底在代碼的哪個位置被修改的。監視的表達式可以是：某個變量的引用、強制地址解析(比如(int )0x12345678，你無法watch一個地址，因為地址是永遠也不會改變的)、合理的表達式(比如a-b+c/d，gdb會檢測其中引用的各個變量)。
　　watch [-l|-location] expr [thread thread-id] [mask maskvalue]
　　thread-id可以設置特定線程改變expr的時候觸發中斷，默認情況下針對硬件模式的檢測點所有的線程都會檢測該表達式；-location會讓gdb計算expr的表達式，並將計算的結果作為地址，並探測該地址上的值(數據類型由expr計算結果決定)。
　　很多平台都有實現監視點的專有硬件，如果沒有GDB也會採用虛擬內存技術來實現監視點，而如果前面兩種技術都不可用，則GDB會採用軟件實現監事點本身。軟件模式速度會非常慢，因為如果不支持硬件模式gdb會每次step並計算檢測的表達式，x86構架支持硬件模式，而且監視點的作用也有限，只能當前單個線程能偵測其變化，雖然該值也可能會被其他線程修改。
　　監視點在跟蹤那種變量不知道哪裡被修改的情形特別的有效。不過監視點需要變量在當前上下文可訪問的情況下才可以使用，所以在使用中會通常配合斷點先到達事發的附近位置，然後再創建對應的監測點。
　　watch命令還存在兩個變體：rwatch當expr被程序讀的時候觸發中斷；awatch會在程序讀取或者寫入expr的時候被中斷。rwatch和awatch只支持硬件模式的檢測點。
　　info watchpoints
　　查看檢測點信息，和短線信息輸出格式類似。
　　當檢測點舊監測的變量或者表達式超過了其作用域的時候，gdb會自動刪除失效的觀測點，所以當跳出變量的作用域後又進入的話，需要重新創建檢測點。尤其當調試程序重啟的時候，所有局部變量的觀測點都會消失，只有全局變量的檢測點會保留。

2.1.3 捕獲點

　　可以讓gdb在某些事件發生的時候暫停執行，比如C++拋出異常、捕獲異常、調用fork()、加載動態鏈接庫以及某些系統調用的時候，其格式為catch event，還有一個變體tcatch event設置臨時捕獲點，其中event的參數可以為：
　　throw|rethrow|catch [regex]
　　在C++異常拋出、重新拋出、捕獲的時候觸發，可選使用regex參數限定特定的異常類型(在gcc-4.8開始支持)，內置變量$_exception會記錄在catchpoint激活時候的異常。
　　當異常發生時候，程序通常會停留在libstdc++的異常支持點，此時可以通過使用up命令切換幀跳轉到用於異常代碼處。
　　syscall [name | number | group:groupname | g:groupname] …
　　在進入和/或返回系統調用的時候觸發。name可以指明catch的系統調用名(定義在/usr/include/asm/unistd.h，且gdb會幫助智能補全)，group|g:groupname可以用來指定一組類別的系統調用，比如g:network，通過智能補全可以查看支持的group信息。
　　exec|fork|vfork
　　load|unload [regex]
　　加載和卸載共享庫時候觸發，可選regex進行過濾。
　　signal [signal… | 『all']
　　可以在軟件收到信號的時候觸發。gdb本身會佔用SIGTRAP和SIGINT兩個信號，如果不添加額外參數，會catch除了這兩個信號之外的所有信號。
　　使用info break命令，watchpoint的信息會被展示出來，可以像普通斷點一樣管理之。

2.1.4 斷點管理(刪除、禁用)

　　clear
　　clear用於清除GDB將執行的下一個指令處的斷點，另外一種表述方法就是在最內棧中，可以用來刪除正在被暫停的這個斷點。
　　clear location
　　刪除指定位置處的斷點，可以是func、file:func、linenum、file:linenum。
　　delete [breakpoints] [range…]
　　參數breakpoints可用可不用，可以指明breackpoints、watchpoints、catchpoints的值或者範圍來刪除它們，他們的編號在同一編號空間中。
　　disable|enable [breakpoints] [range…]
　　禁用|啟用某些斷點，如果沒有指明範圍，則是針對所有的斷點。
　　enable [breakpoints] once range…
　　臨時啟用某些斷點，一旦這些斷點激活，就會被自動disable。
　　enable [breakpoints] count cnt range…
　　這種情況下臨時使能的斷點，會在每次被激活的時候遞減cnt計數，當其值為0的時候，會禁用該斷點。這個count還有一個好用的方式是和ignore相結合，可以在循環中跳過cnt個循環。
　　enable [breakpoints] delete range…
　　臨時啟用某些斷點，一旦這些斷點激活，就會被自動delete，跟通過tbreak設置的斷點起始狀態一樣。

2.1.5 條件斷點

　　當指定的cond為true的時候，斷點就會被觸發，其可以用於普通斷點，也可以用於watchpoint(雖然值一旦改變就會觸發，但是可以用cond過濾得到感興趣的值)。條件斷點可以具有side-effect，比如其可以調用函數、寫入日誌、執行命令等操作，副作用是可控的，除非該條件斷點上又設置了別的斷點，而別的斷點使程序暫停此時該cond可能就沒有被檢查調用。
　　普通斷點和檢測點可以在命令break/watch的時候使用 if關鍵字指明條件，但是catch不能識別if關鍵字；其上所有的斷點都可以在創建之後使用condition關鍵字指明條件來創建和刪除條件：
　　condition bnum expr
　　創建和刪除斷點的條件，在expr為true的時候斷點才會被觸發。
　　條件斷點的表達式幾乎可以是任何類型，可以是庫函數、甚至是自己的函數(該函數需要被鏈接到程序中)。不過需要注意的是，如果調用了庫函數，但是庫函數沒有調試符號(大多數髮型庫的行為)，那麼將其使用到條件中時該庫函數的返回類型只假定為int，而且強制轉換也不行，這一限制不僅在條件表達式中，在GDB任何表達式中調用庫函數都是這樣的。
　　condition bnum
　　刪除條件後斷點，退化成非條件斷點。
　　ignore bnum cnt
　　讓斷點可以忽略cnt次後生效，如果是使用continue命令恢復程序的執行，也可以使用continue cnt的方式代替ignore指令。如果一個斷點有ignore和condition兩種屬性，那麼在ignore到達0之前是不會檢查condition條件的。

2.2 繼續和單步執行

　　continue|c|fg [ignore-count]
　　從程序最後停止的位置繼續執行，且該位置上的所有斷點將會被bypassed。可選的ignore-count和ignore作用相同，該參數隻有在當前停止是由於斷點觸發的時候才有效，否則該參數將會被忽略。
　　如果想從別的地方繼續程序的執行(而不是當前暫停的位置)，可以選用return到函數的調用位置、jump到任意位置開始恢復執行。
　　step|s
　　繼續執行當前的程序直到達到一個新的代碼行號後，停止程序並將控制轉給GDB。step只會停留在代碼行的第一條指令處，如果在當前行中有調用帶有調試符號的函數，那麼step會繼續停留，即step會進入到帶符號的函數中去，否則其行為類似於next命令而不會進入到函數中去。
　　說到這裡，需要注意的是step命令只會停留到帶有調試符號的代碼中，如果在沒有調試符號的函數環境中使用step，那麼程序會一直執行直到到達一個含有調試信息的函數；否則，需要stepi命令。
　　step count
　　持續執行count次的step，如果在這之前遇到一個斷點或者和step無關的信號，step將會停止。
　　next|n [count]
　　在當前(最內層)的stack frame中執行到下一個代碼行，和step不同的是代碼行中的函數調用將會被直接執行而不會停止。類似的，next也只會在代碼行的首個指令處停止執行。
　　finish|fin
　　繼續執行直到當前選定的stack frame的函數返回，如果可以的話就打印函數的返回值，簡言之就是將當前函數執行完。finish的一個常見用途就是不小心跟蹤到一個函數中間去(比如原本應該用next的時候使用了step)，此時就可以使用finish帶你到原本next應該停留的位置上去。
　　和這個命令對應的是return [expr]命令，該命令是取消當前函數的後續執行直接返回，並且可選返回一個表達式作為調用結果。
　　until|u
　　繼續執行直到在當前stack frame中到達超過當前行號的位置(當然也可以手動使用臨時斷點達到類似效果)，這個命令在循環中使用的比較多，而在非循代碼下跟next命令比較相似，而且until命令在退出當前stack frame的時候總是會停止程序的執行。手冊也表述為執行程序直到達到當前循環體外的下一行代碼，實際是達到內存地址比當前地址更高的機器指令，同時注意的是，編譯器為了效率可能會重排循環代碼，比如GCC使用循環主體底部條件編譯程序，這時候until的行為可能會有些詭異，多執行一次until就可以達到目的了。
　　until|u location
　　相比不帶參數的until效率會更高(採用了臨時斷點方式實現)，會執行程序直到特定位置或者退出當前stack frame，所以只有location是在當前stack frame內部的時候才會到達，location可以是行號、地址等任意類型的位置信息。
　　該until命令的另外一個好處，是可以跳過(skip over)遞歸函數調用。

　　如果當前執行點在第5行，until 7命令會持續整個遞歸調用，直到value=120的時候在第7行暫停。
　　advanced location
　　和until比較類似，但是不會跳過遞歸調用，同時location的位置也不需要限定在當前的stack frame。
　　stepi/si count
　　執行一條(count條)機器指令，在使用這種方式調試的時候，建議使用display/i $pc顯示機器指令的彙編代碼。
　　nexti/ni count
　　和上面類似，只是在遇到函數調用指令的時候不會跟蹤函數調用，會直接等待函數返回。

2.3 跳過函數和代碼文件

　　有些時候調試中可能對某個函數、某個文件中的所有函數或者特定文件中的特定函數不感興趣。
　　比如：foo(boring());如果不想調試boring()函數，方法是先step進入boring()，然後使用finish跳出該函數，再繼續調試foo()函數。使用skip命令可以設置忽略某些函數的調試達到上述的效果。
　　skip function [linespec]
　　執行該命令後，linespec指定的函數名或者linespec行號所在的函數將會被跳過。如果沒有指定linespec，那麼當前正在被調試的函數名將會被作為參數使用。
　　skip file [filename]
　　執行該命令後，所有源代碼在filename中實現的函數，都會在stepping的過程中被跳過。如果沒有指定filename，那麼當前調試焦點所在的源代碼文件將作為其參數。
　　skip [options]
　　是一個基本更靈活的命令(不過測試發現極大發行版的gdb不支持該模式)，通過-file|-function可以得到上面另個子命令相同的效果，而且可以任意組合使用。更強大的是使用-gfile|-rfunction的正則表達式模式，來指定符合正則表達式的文件後者函數名。

　　如果沒有任何參數，其效果類似於命令skip function。
　　info skip
　　skip enable/disable/delete [range]
　　常規的skip管理命令。

2.4 信號

　　信號是操作系統定義在程序中可以異步發生的事件，有些信號是程序正常執行的一部分(比如SIGALRM)，而有些信號預示著錯誤甚至是致命的(比如SIGSEGV)，通常會終止程序的執行。很多信號如果沒有事先設置好處理handler，默認的處理方式是終止該程序的執行。額外提一下，Linux下進程的內存佈局可以通過訪問/proc/pid/maps查看。
　　GDB可以感知程序中任何信號的發生，可以設置對特定信號的特定處理方式。正常情況下GDB是讓非錯誤的信號(比如SIGALRM)傳遞到調試的程序，而讓錯誤信號發生的時候停止程序的執行(具體可以使用info signals查看)，可以通過handle命令修改特定信號的行為。
　　catch signal [sig…|』all']
　　給對應信號設置catchpoint。
　　handle sig [keywords…]
　　修改GDB對信號的默認處理方式，sig參數可以是SIG打頭的信號名、low-high的信號值範圍、』all'所有信號；keywords可以是下面的關鍵字
　　nostop、stop：當該信號發生的時候gdb是否要停止程序的執行；stop->print
　　print、noprint：當該信號發生的時候是否打印相關信息；noprint->nostop
　　pass|noignore、nopass|ignore：GDB決定是否讓你的應用程序看到該信號，即應用程序消費該信號。對於非錯誤信號的默認模式是nostop|noprint|pass，對於錯誤信號的模式是stop|print|pass。
　　當一個信號停止了程序的執行，被調試的應用程序在continue之前是覺察不到該信號的，在GDB報告一個信號的發生的時候，可以通過handle命令的pass|nopass來設置是否傳送該信號，然後繼續程序的執行。
　　GDB對於調試中的信號處理做了一定的優化：
　　如果信號是nostop的，此時使用stepi/step/next進行跟蹤調試的時候，將會優先執行信號處理函數，在信號處理函數返回後再執行主線調試代碼，即會step over信號處理。不過信號處理函數本身也可能有斷點等因素導致執行流過早的停止。
　　當使用了stop的時候，並且程序設置了對應信號的信號處理函數，此時程序如果因為信號暫停的話，則使用step/stepi/next將會跳入到信號處理函數中去跟蹤。

2.5 多線程程序

　　在多線程程序調試中，具有兩種工作模式：all-stop和non-stop。

2.5.1 all-stop mode

　　當進程中任意線程因為某種原因停止執行的時候，所有其他線程也同時被GDB停止執行。好處是程序停止的時候所有線程的狀態都被保留下來了，可以切換線程查看整個程序當時的所有狀態(缺點是整個業務全部阻塞暫停了)。
　　在默認設置下，當恢復程序執行的時候，所有的線程也都會立即執行，即使是使用step/stepi/next這種單步跟蹤的模式也是如此。線程的調度是操作系統控制的，即使單步跟蹤的時候很有可能其他的線程執行了更多指令後，當前線程才執行完單步指令並停止；還有可能單步跟蹤後發現自己停止在了其他的線程上面，因為任何時候任何線程因為斷點、信號、異常的時候都會停止程序的執行，並且GDB會自動選定切換到該線程上面去(同時會打印Switching的切換信息)。
　　有些操作系統支持OS調度鎖定的支持，就可以修改上述GDB的默認行為。
　　schedule-locking
　　off：不會鎖定，所有線程可以自由運行；
　　on：當程序resume的時候，只有當前線程可以運行；
　　step：該模式是為單步模式優化的模式，當跟蹤的時候阻止其他線程搶佔當前線程。當step的時候其他線程無法運行，而在使用continue、until、finish類似指令的時候其他線程可以自由運行。除非其他線程運行時候觸發了斷點，否則GDB不會切換當前調試的線程。
　　schedule-multiple
　　該設置是針對多進程情況下的調試。
　　off：只有當前線程所在的進程的所有線程允許恢復執行，該off為默認值。
　　on：所有進程的所有線程都可以執行。

2.5.2 non-stop mode

　　當進程中任意線程停止時，在跟蹤檢查停止線程的時候其他的線程任然繼續執行，其好處就是對線上系統進行最小化的干擾入侵，整個程序還是可以繼續響應外部請求的。
　　通過下面的設置可以開啟non-stop模式，一般都是在gdb開始的時候或者連接調試目標的時候設置才會生效，且在調試過程中不能切換該模式，不是所有平台都支持non-stop模式，所以即使設置了GDB還是有可能fallback到all-stop模式。

　　non-stop模式的驗證也十分的簡單，在一個多線程的網絡程序中先設置斷點，此時發送一個請求那麼處理線程會被停止，接下來刪除該斷點，再次發送一個請求，看該請求是否得到正常響應。如果non-stop正常開啟生效的話，此時的info threads會顯示其他線程都是running的狀態。
　　non-stop模式下的命令默認都是針對當前線程的，如果要針對所有線程，通常需要使用-a參數，比如continue -a。

2.5.3 線程相關的斷點

　　break location thread thread-id [if …]
　　在多線程模式下，可以設置線程相關的斷點(而不是默認針對所有線程)，其語法類似，只不過需要使用thread關鍵字指明GDB分配的線程標識符。當線程退出或者gdb在detach的時候，該線程相關聯的斷點會被自動刪除。

2.5.4 中斷的系統調用

　　使用GDB調試多線程程序的時候有一個副作用，就是當線程因為斷點或其他因素暫停的時候，其他線程此時如果阻塞在系統調用上，此時的系統調用可能會過早的退出！因為多線程程序調試的時候線程之間需要可能會用信號等手段來通信。
　　對此，應當培養良好的系統調用使用習慣，多做系統調用返回值的判斷。

三、程序的執行的回放

　　GDB提供process record and replay target(後面簡稱R&RT)的功能，可以記錄程序的執行過程(execution log)，並在後續時間上進行正向、逆向的程序運行。在調試目標運行的時候，如果接下來的執行指令在log中存在，就會進入replay mode，該模式下不會真正的執行對應的指令，而是將事件、寄存器值、內存值直接從日誌中提取出來；如果接下來的執行指令不在log中，GDB將會在record mode執行，所有的指令將會按照正常方式執行，GDB還會記錄log方便將來reply。
　　進程的R&RT支持正向和逆向執行，即使對應的平台底層調試不支持reverse exec，但是通過reply mode可能模擬這個功能，而record mode可能由於平台的限制不支持reverse exec。
　　record *method
　　開啟進程的R&RT模式。默認的方法是使用full recording，其實際是GDB採用軟件模擬的方式實現的，該模式支持replay reverse exec，不過不支持non-stop mode和異步執行模式的多線程調試。R&RT只能調試正在執行的進程，所以事先需要先使用run|start命令開啟調試進程，然後使用record method進行記錄。
　　record stop
　　停止進程的R&RT，此時所有的log將會被刪除，而調試進程將會被終止揮著保持在其最終狀態。
　　如果是在record模式下(在log的最末尾)使用該命令，被調試的進程將會在下一個要被記錄的指令處停止，即如果record了一段時間後停止record，那麼調試進程將會保持在類似record從未發生過的狀態；如果在replay模式(就是在log的某個中間位置而非末尾)下執行該命令，那麼被調試的進程將會在那個點開始進行live正常調試；如果被調試進程退出，或者被detach，那麼R&RT將會自動停止。
　　record goto begin|start|end|n
　　跳轉到指定的log位置，除了特殊的begin|start|end，其中的n是log中的instruction的標號。
　　record save|restore filename
　　log的保存和恢復。如果不指明文件名將會是默認的』gdb_record.process_id'。
　　set record full insn-number-max limit|unlimited
　　設置log記錄的指令的數目，默認值是200,000。如果設定的limit值是個正數，記錄的指令數目超過這個限制後GDB會依次從最先的指令中刪除這些舊指令以在不超過這個限制的條件下保存新指令記錄，如果其參數值是0或者unlimited，那麼久不會有保存數的限制，最終保存的多少將受內存大小的限制。
　　set record full stop-at-limit on|off
　　設置在full方式下當保存指令數目超過限制的生活，如果ON(默認)，則GDB會在首次超過限制的生活停止調試進程，並向用戶詢問是否繼續執行下去；如果是OFF，則GDB會自動進行上面刪除最舊記錄添加最新記錄的行為。
　　info record
　　顯示record記錄相關信息，包括當前是Record mode還是Replay mode、最小最大指令記錄、當前保存的指令數目、可記錄最大條目數限制值等信息。
　　record delete
　　當在reply模式下執行該命令的時候，會刪除當前所在所在指令之後的所有log記錄，即丟失當前指令之後的所有記錄，並開始新的record。

四、棧信息檢查和回溯

　　由函數調用的原理，一個運行的程序會有多層的stack frame，當被調試程序停止的時候GDB會默認選中當前執行的stack frame(當然後面可以手動選擇在各個stack frame之間選擇切換)，後續的操作都隱含針對當前選中的stack frame。

4.1 棧結構

　　棧幀(stack frame)包含的是傳遞給函數的參數、函數中的局部變量、已經函數執行的地址(返回地址)信息。當程序最開始執行的時候，只有main函數的一個frame，稱其為initial frame或者outermost frame，frame會在函數調用的時候創建並在函數返回的時候刪除，當前正在執行的棧稱為innermost frame，其地址由frame pointer register寄存器保存，從而可以方便的訪問調用參數和局部變量。
　　GDB會從innermost開始upward，由0開始對frame進行編號，以方便對特定的stack frame進行引用。
　　某些編譯器提供某種方法使得編譯出來的函數沒有stack frame，比如gcc的-fomit-frame-pointer選項，為了就是讓那些大量調用的函數節省frame的建立時間，不過其缺點是棧回溯變得困難了。

4.2 Backtrace

　　從當前執行的frame(innermost, 0)開始，沿著其調用者upward的過程。
　　backtrace|bt [n|-n]
　　如果沒有參數，會打印整個stack，每一行代表一條frame；如果n表示打印innermost n的frame；如果-n表示只打印最outermost n的frame。
　　命令info stack是backtrace的別名。
　　backtrace|bt full [n|-n]
　　參數和上面的類似，但是還會同時打印每frame的局部變量(調用參數會直接在frame的那一行顯示出來)。
　　
　　在多線程程序的調試中，GDB默認只對當前選中的線程顯示backtrace，如果要對其他的線程同樣執行類似的跟蹤，需要使用前綴的方式thread apply all backtrace才能達到效果，尤其對於調試多線程程序的coredump十分有效。
　　backtrace中的函數參數、變量的值可能顯示不完美，對於非標量類型的變量會用…顯示，而有限變量可能會被優化掉(比如調用的過程中使用寄存器傳參)，此時其值會被用』『代替。
　　默認棧完全回溯會跟蹤到main這個outermost層次，如果需要跟蹤啟動代碼，可以設置set backtrace past-main on變量，不過我想一般都用不著這麼個層次吧。
　　默認情況下backtrace是沒有層次限制的(一直頂到main)，通過set backtrace limite n|0|unlimited可以進行設置。

4.3 選擇棧的frame和查看frame

　　上面說過，很多命令都是隱含針對當前選定的frame操作的。因為GDB對frame都進行了編號，所以既可以用編號進行引用選擇，也可以採用相對的up/down操作進行frame的切換。
　　frame|f n
　　使用frame編號進行選擇，0代表最innermost(當前正在執行)的棧，最大值就是main frame的編號值。
　　frame|f stack-addr
　　在stack-addr處選擇frame。這通常在frame因為bug被破壞了，導致GDB無法為frame進行編號的生活，以及當程序具有多個stacks並且在他們之間切換的時候。
　　up|down|up-silently|down-silently n
　　相對形式的切換frame，參數n默認為1，其值也可以為正數或者負數，遍歷調用棧可以方便查看各個棧的信息，而不會影響程序的執行路徑。

　　frame|f
　　該命令不會切換當前選擇的幀，而是簡潔的打印當前幀的一些信息。
　　info frame [addr]
　　更加詳細的打印當前選中frame的信息。
　　info args|locals
　　前者打印當前frame的調用參數，後者顯示所有可訪問的局部變量(包括static或者automatic聲明的)。

5.2.1 行表示方法

　　filename:linenum

　　function

　　-source filename
　　通常單獨的source沒有意義，它是和下面的function、line等結合使用的。為了消除可能的文件名歧義，建議對於常用的文件名增加一些額外的路徑前綴。
　　-function function
　　-label label
　　-line number
　　這裡的line既可以是絕對的行號，也可以是相對的(+|-3)行號位置。比如break -s main.c -li 3。

3.2.3 地址信息位置表示

　　funcaddr
　　*』filename':funcaddr

5.4 源代碼和機器碼的對應及反彙編

　　通過info line可以建立源代碼行號和程序地址之前的對應關係，而disassemble可以顯示一段地址中機器指令。GDB還有一個變量』disassemble-next-line'，當打開他的時候會在程序停止的時候自動顯示要執行的下一行源代碼的第一條彙編代碼，而如果下一條執行的指令沒有調試信息，就會顯示下一條機器指令的反彙編結果。
　　info line location
　　打印locatin代表的源代碼行編譯後的其實地址和結束地址。當然也可以進行反方向的隱射查詢，通過地址查詢其對應源代碼的那一行

　　disassemble [/m|/s|/r] [start, end]|[start, +len]
　　disassemble [/m|/s|/r] location
　　可以顯示一段地址範圍的反彙編信息，或者函數名等其他代表的地址信息的反彙編結果，如果是跨文件的記住使用disassemble 『foo.c'::bar這樣的形式。/m或者/s參數可以顯示源代碼和反彙編代碼同時交叉顯示的效果，/r還會以hex的方式顯示機器碼，當然這對一般人來說使用較少。
　　set disassembly-flavor intel|att
　　反彙編是顯示intel還是att的風格，個人比較偏向att的格式，這也是GDB的默認行為。

六、查看數據　　

　　最常用的檢查數據的命令就是print|p|inspect，其接受一個表達式作為參數，會根據調試的語言對表達式進行求值，並將其結果打印出來。

　　一種更加底層檢查數據的方式是使用x命令，該命令會檢查內存中特定地址的數據並通過特定的格式顯示出來。
　　如果想知道表達式的類型而不是關係其值的話，可以使用ptype expr命令，就可以顯示出struct、class等複雜變量的類型信息。

6.1 表達式

　　GDB的表達式可以支持普通表達式、條件表達式、函數調用、類型轉換、字符串常量、宏等類型，同時使用』{elem1, elem2, …}』這種表達式還可以創數組常量。
　　@ 該操作符用於將一段內存當做數組使用
　　:: 該曹組福用於指明在文件或者函數中的變量
　　{type} addr addr可以是一個整形或者指針，該表達式用於將指定位置的內存引用為type類型的變量。

6.2 二義性的表達式

　　由於重載和模板機制，在C++這類語言中二義性的表達式非常常見，在這種情況下通常需要給出函數的參數簽名』function(types)』等信息才能和其他同名對象進行區分。GDB會自動探測二義性的存在，給出所有可能性的選擇列表，並最終給出』>』提示符信息讓用戶選擇，固定的選項是』[0] cancel'和』[1] all'，後面的選項是各個明細，比如break String::after這樣的命令，用空格分割的選項才會打上斷點。
　　set multiple-symbols all|ask|cancel
　　默認參數值是all，當某個命令的表達式有多個選項的時候，GDB會自動選擇所有可能的選項，但是某些環境下必須有一個選項的時候，GDB會顯示出上面所述的菜單供選擇；ask表示只要二義性檢測到了，就會給出菜單讓做出選擇；cancel表示當debuger探測到二義性的時候會當做錯誤，相應的命令被放棄執行。

6.3 程序變量

　　程序的變量算是最常被在表達式中使用的，變量是在當前選擇的stack frame中可見的，他們可以是：全局變量、file-static變量、根據變成語言語法規則可見的變量。這裡有一個例外，就是你可以引用file-scope的變量或者函數，即使當前執行的上下文不在該文件當中。
　　各個函數中的變量，以及各個文件中的函數或者變量可能會出現重名的情況，這個時候就需要』::』操作符進行指定

　　在調試的時候，有時候會出現在函數進入作用域和返回的生活，某些局部變量的值是錯誤的，尤其是當進行機器指令單步的時候，因為很多機器上stack frame的建立(包括局部變量的定義)和函數返回的時候stack frame的銷毀，都不是單個機器指令可以解決的，所以在stack frame完全建立和銷毀的時候，檢查到的變量值很可能是異常的。
　　一些沒有被使用的變量也可能被編譯器優化掉，這時候價差這些變量的值可以提示他們不存在。

6.5 輸出格式

　　print默認是根據表達式的類型自動顯示的，在大多數情況下都可以工作的很好，但是在print中可以指定輸出格式得到個性化得顯示結果。下面的格式參數通過/和print命令相連，比如print/x $pc

6.6 內存檢查

　　通過命令』x'可以進行內存數據檢查。其命名格式為(nfu是控制內存顯示的大小、格式等參數信息的)
　　x /nfu addr
　　n(顯示數目) 默認值是1，指明需要單位內存(u指明)顯示的數目。如果n的值是負數，那麼將會從addr開始進行反方向的內存顯示。
　　f(顯示格式) 顯示的格式參數和上面print命令的輸出格式相類似(包括：x、d、u、o、t、a、c、f、s)，初次還包括i用於機器指令的顯示。默認情況下是使用x進行hex顯示，該值會隨著x|print命令的操作而更新。
　　u 顯示單位尺寸 b(bytes，1字節)、h(半字，2字節)、w(字，4字節)、g(gaint字，8字節)，同樣該參數會隨著每次使用x被自動記住並在下次命令中自動應用。對於i顯示機器指令該參數會被忽略，而對於s該參數默認是b。
　　由於f和u兩個格式參數的空間是不重疊的，所以可以以任意順序顯示，但是n必須在他倆之前。
　　addr 指定需要檢查的起始地址位置，比如函數名等，其最終會被解析成一個整形的地址值。

6.7 自動顯示

　　對於需要經常查看的表達式，可以使用display命令將其添加到自動顯示列表中，這樣每次程序停止執行的時候，就會自動顯示該表達式的值，加入的表達式會自動產生一個遞增的整形編號。display也可以使用print|x命令的參數進行格式化輸出控制，而且其內部如果發現格式化參數包含i|s或者數字，就會調用x；否則會調用print進行輸出。
　　display[/fmt] expr
　　display/fmt addr 使用』i'、』s'格式，或者指定了數字
　　將表達式或者要檢查的內存地址添加到自動顯示列表中去。
　　比如想要自動顯示下一條要執行的指令，可以做如下添加：display/i $pc。
　　undisplay dnums…
　　將對應編號的表達式或者內存從自動顯示列表中刪除。列表可以使用逗號』,』進行分割，以及使用2-4這種範圍表示。
　　info display
　　enable|disable display dnums…
　　查看、禁用或者使能自動顯示。
　　display
　　進行自動顯示列表中表達式值的顯示，就像程序剛停止時候自動顯示的效果相同。
　　如果添加自動顯示的表達式引用到了局部變量，GDB會自動考慮其上下文信息，如果執行的上下文中該局部變量沒有定義則不會顯示其值，而如果下次再次進入定義該變量的上下文，則該表達式變得有意義會再次顯示其值。無論如何，info display總會顯示該表達式的相關信息。

6.8 數值歷史(Value History)

　　通過使用print命令顯示的值會被自動保存在GDB的數值歷史當中，該值會一直被保留直到符號表被重新讀取或者放棄的時候(比如使用file或symbol-file)，此時所有的值歷史將會被丟棄。在使用print打印值的時候，會將值編以整形的歷史編號，然後可以使用$num的方式方便的引用，單個的$表示最近的數值歷史，而$$表示第二新的數值歷史，$$n表示倒數第n新的數值歷史(所以可以推斷$$0==$; $$1==$$;)。
　　比如剛剛打印了一個指向結構體的指針，那麼print $就可以顯示結構體的信息，而命令print $.nex甚至可以顯示結構體中的某些字段。

6.9 Convenience變量

　　GDB允許自由創建便捷變量用於保存值，後續可以方便的引用該值，該類型的變量由GDB管理而與正在調試的程序沒有直接的關聯。便捷變量也是使用符號$打頭的，可以使用任意名字(除了GDB使用的寄存器名)。
　　在任意時候使用一個便捷變量都會創建他，如果沒有提供初始化值那麼該變量就是void，直到給其進行賦值操作為止。便捷變量沒有固定的類型，可以為普通變量、數組、結構體等，而且其類型可以在賦值過程中改變(為當前值的類型)。
　　show convenience|conv
　　顯示道目前所有的便捷變量和便捷函數，以及其值。
　　init-if-undefined $variable = expr
　　如果該變量還沒有被創建或初始化，則創建這個變量。如果該變量已經被創建了，則不會創建和初始化該變量，並且expr表達式也不會被求值，所以這種情況下也不會有expr的副作用發生。
　　便捷變量的一種經典用法，就是之前提到的連續查看變量時候用於計數作用：

　　下面的一些便捷變量是GDB自動創建的：
　　$_exitcode
　　當調試程序終止的時候，GDB將會根據程序的退出值自動設置該變量，並且將$_exitsignal變量設置為void。
　　$_exitsignal
　　當調試中的程序因為一個未捕獲信號而終止，此時GDB會自動將變量$_exitsignal設置為該信號，同時重置變量$_exitcode為void。

6.10 Convenience函數

　　便捷函數和便捷變量一樣使用$打頭引用，其可以像一個普通函數一樣在表達式中使用，便捷函數只被GDB內部使用。

　　$_isvoid (expr)
　　如果expr是void則返回1，否則返回0。
　　GDB還有很多的便捷函數支持，但是需要在編譯GDB的時候提供Python的支持才可以使用。下面的這些函數意義顯而易見，就不在囉嗦了。
　　$_memeq(buf1, buf2, length); $_regex(str, regex); $_streq(str1, str2); $_strlen(str)

6.11 寄存器

GDB可以像普通變量一樣引用機器的寄存器值，通過$打頭。
　　info registers
　　info all-registers
顯示機器所有寄存器的名字和其當前值信息。後者比前者會多出浮點寄存器和向量寄存器。

6.12 操作系統信息

通過直接的info os可以顯示出GDB支持的操作系統信息類別。
　　info os infotype
　　files 列出目標上面打開的文件信息。
　　modules 列出目標上加載的內核模塊信息。
　　msg 列出目標上所有的Sys V消息隊列信息。
　　processes 列出目標上所有的進程信息。
　　semaphores 列出目標上所有Sys V信號量信息。
　　shm 列出目標上所有Sys V共享內存信息。
　　sockets 列出目標上所有Internet-domain socket信息。
　　threads 列出目標上所有線程信息。

6.13 內存和文件拷貝

　　通過dump、append、restore命令可以在目標機之間傳遞數據和內存信息。dump命令和append命令將數據寫入(追加)到文件中，而restore會讀取文件內容並將其加載到調試進程的指定內存位置中。GDB默認使用binary的文件格式，推薦使用該格式。
　　dump|append memory filename start_addr end_addr
　　dump|append value filename expr
　　restore filename binary bias start end
　　其中一個比較重要的參數是bias，如果其值為非0，那麼這個值將會作為偏移添加到文件中所有地址值上面。因為二進制文件總是從地址0開始，所以在恢復的時候需要添加這個偏移值。

6.14 產生core文件

　　core文件是運行進程的內存鏡像和運行狀態信息(比如寄存器等)的集合。通過配置，操作系統可以在程序掛掉的時候自動產生coredump文件，而在GDB調試的時候，也支持使用命令方式手動創建coredump文件，這樣就可以快速創建一個程序在當時的快照。
　　generate-core-file|gcore [file]
　　如果沒有提供名字，那麼默認將產生core.pid文件名格式的dump文件。

6.15 內存查找

　　使用命令find可以對內存進行序列字節的查找操作。
　　find [/sn] start_addr, +len, val1 [, val2, …]
　　find [/sn] start_addr, end_addr, val1 [, val2, …]
　　該命令會在內存中搜索var1、var2…字符序，其範圍從startaddr開始，以結束地址或者長度結束。
　　參數s表示搜索的大小，可以是b、h、w、g(跟之前的x命令一樣)，如果該參數沒有指定，GDB會考慮當前調試程序的語言環境確定，比如0x43這種整數就默認為w(4字節)類型；n表示打印最大搜索命中數量，默認行為是全部打印。該命令還可以搜索字符串，字符串使用雙引號括住。
　　對於查找結果，命中數目保存在變量$numfound中，而最後一個值命令的地址保存在**$**中。

七、跟蹤點(Tracepoint)

　　有些情況不適合把線上的業務停下來，進行長時間的跟蹤和調試；還有些行為只有在線上實時運行的環境才能重現，延時或者線下環境無法復現的問題，針對這種類型的情況，需要不打斷程序的運行去觀察程序的行為。
　　通過GDB的trace和collect命令，可以在程序中設置跟蹤點，然後在跟蹤點到達(命中hit)的時候計算事先構造的表達式的值；接下來，通過tfind命令，可以檢查這些跟蹤點中表達式在當時所記錄下來的值。當前跟蹤點的功能值在remote target的情況下支持，該功能是在remote stub中實現的，但是當前情況下的GDB還沒有實現，目前主要使用的方式是使用文件的方式操作。

7.1 設置跟蹤點

　　跟蹤點實際上是一種特殊的斷點，所以可以使用任何標準斷點的命令來操作跟蹤點，和斷點不同的是不支持ignore count、不支持針對特定於線程的跟蹤點。對於每一個跟蹤點，可以事先指明到達的時候所需要蒐集的數據，包括寄存器、局部變量和全局變量，之後可以使用GDB的命令來檢查這些在當時收集到的數據值。
　　跟蹤點是動態跟蹤工具，意味著這些跟蹤點可以被插入在目標的任何位置。
　　靜態跟蹤點暫且不表。

7.1.1 創建和刪除跟蹤點

　　trace location
　　**其中的location可以是和break命令參數一樣的任意有效位置，通過trace命令可以創建一個跟蹤點，當debuger遇到的時候會暫停，收集完相應地數據後繼續執行。GDB還支持pending tracepoint，類似於斷點的原理，表示在一些共享庫還沒有加載的時候地址無法解析。

　　trace location if *cond
　　帶有條件的，只有當cond取值為true(非零)的時候，才會進行數據的收集，避免每次到達跟蹤點都收集數據導致數據量暴增，而往往很多情況下的數據是沒有價值的。
　　條件跟蹤點既可以在創建的時候使用if進行限定，也可以後續使用類似breakpoint情況下的condition命令進行設定修改。和斷點不同的是，GDB不會自己去對條件表達式進行取值，而是將該表達式編碼給agent(獨立於GDB)。
　　delete tracepoint [num]
　　永久性地刪除一個或者所有(不帶參數)的跟蹤點。
　　disable|enable tracepoint [num]
　　禁用|使能某個或者所有的跟蹤點。
　　info tracepoints [num…]
　　顯示某個或者全部跟蹤點的信息，使用方法和info breakpoints相同。

7.1.2 跟蹤點計數

　　passcount [n [num]]
　　passcount是一種自動停止trace experiment的方式，當tracepoint被命中n次之後trace experiment會自動停止。
　　當使用passcount帶有n但是不帶num參數的時候，n將會作用於最近添加的跟蹤點。如果不使用passcount進行相關設定，默認行為的跟蹤點將會一直運行直到被用戶顯式地停止。

7.1.3 跟蹤狀態變量

　　trace state variable是一種特殊類型的變量，它是在target-side創建和管理的(但是對GDB可見)，和GDB的convenience變量語法相同，必須採用tvariable命令顯式創建，並且總是64位的整形。
　　雖然trace state variable是被target端管理的，但是可以在GDB中像使用convenience variable一樣使用它們，GDB會在trace experiment運行的時候從target端獲取該變量的當前值。該變量也是使用符號$打頭的，和GDB其他變量在同一個名字空間中，所以需要注意不能重名。
　　tvariable $name [ = expr]
　　創建名為$name的trace state variable，並且可選地給其一個初始值。初始值得表達式expr是在該變量創建的時候就進行取值的，得到的結果將會轉換成64位整形類型。後續再使用tvariable接上已經存在的變量名的時候，實際不會進行創建操作，而是將之前的變量進行重新賦值。如果沒有提供初始值，默認初始值是0。
　　info tvariable
　　delete tvariable [ $name … ]
　　如果沒有提供額外參數，將會刪除所有的變量。

7.1.4 跟蹤點行為列表

　　actions [num]
　　用於指明在跟蹤點被命中的時候所需要執行的操作，如果沒有提供num參數，那麼該設置默認針對於最新創建的跟蹤點。
　　這些行為列表中的命令一行一個，通過單個end標示著結束，當前所支持的行為包括collect、teval和while-stepping，其實使用起來跟command命令十分的類似，只不過其命令的內容只能是上面允許的，而不能是其他的GDB調試命令。要想刪除跟蹤點的actions，則提供空的end就可以了。

　　collect [/mods] expr1, expr2, …
　　收集用逗號分隔的各個表達式的值。收集的對象除了全局變量、靜態變量、局部變量，還可以包含：
　　$regs 所有的寄存器
　　$args 函數的所有調用參數
　　$locals 所有局部變量
　　$_ret 函數的返回地址
　　teval expr1, expr2, …
　　當跟蹤點被命中的時候會計算表達式的值，該命令接收使用逗號風格的多個表達式。取值表達式的結果將會被丟棄，所以該命令的主要用處是在不添加這些值在trace buffer的同時，將表達式計算並賦值給trace state variable。
　　while-stepping n
　　在跟蹤點之後進行n步的單步調試，並且在每次單步執行後進行數據的蒐集，需要收集的數據在while-stepping後使用collect表示，並使用end結尾。
　　set default-collect expr1, expr2
　　該變量可以是使用逗號分隔的一大串表達式表明默認需要收集的值，表明在每個跟蹤點默認都要蒐集的數據。注意的是這個表達式是在每個跟蹤點獨立解析的，所以相同的表達式、名字在各個跟蹤點可能被解析為不同對象。

7.1.5 開始和停止跟蹤執行

　　tstart
　　該命令啟動trace experiment並且開始蒐集數據，該命令的副作用是丟棄所有在trace buffer中的已經存在的歷史數據。如果給該命令提供了參數，他們將會被視為note並被存儲起來。
　　tstop
　　該命令會停止trace experiment，如果某個跟蹤點已經達到其passcount，或者trace buffer的緩衝區已經滿了，那麼trace experiment會自動停止。如果給該命令提供了參數，他們將會被視為note並被存儲起來。
　　tstatus
　　用於顯示跟蹤和收集到的數據信息。

7.1.6 Tracepoints的限制

　　因為收集數據的行為是在target自動進行的，所以GDB無法做出細緻的控制，而且後續檢查數據的時候也僅基於已經收集到的歷史數據。
　　(1) 因為跟蹤點的首要目標是收集作值，所以GDB的複雜表達式往往是不可用的。在收集的過程中不能調用函數、進行類型轉換、訪問convenience變量、修改變量值等操作。

7.2 使用跟蹤記錄的數據調試

　　當trace experiment結束後，可以使用GDB命令來查看已經收集到的數據，這些收集的數據都是每次命中跟蹤點收集到數據的快照。在檢查數據的時候都會選中一個快照，然後所有GDB的命令(比如print、info registors、backtrace等)的請求，都會像當時跟蹤點命中時候的狀態值進行對應返回，而此時如果請求那些沒有收集到的數據將會失敗。

7.2.1 tfind

　　通過使用命令tfind n可以查看緩衝區中第n個快照，緩衝區中的快照都是從0開始連續遞增形式編號的，gdb每命中一個tracepoint就會產生一個快照，如果tfind你沒有接參數，將會依次訪問下一個快照。
　　tfind start
　　選擇快照列表中的第一個快照，跟tfind 0命令效果等價。
　　tfind none|stop
　　停止基於trace snapshots的跟蹤，恢復在線實施跟蹤。
　　tfind |-
　　如果tfind不帶參數，則繼續選擇相對於當前選擇快照的下一個快照；如果使用-，則表示向前的順序選擇前一個快照。
　　tfind tracepoint num
　　因為snapshot是連續編號的，但是每個snapshot可能對應任意的tracepoint，通過這個命令可以選擇對應於編號num的tracepoint的下一個快照，可以將其想像成連續跟蹤特定tracepoint的變化狀態，如果參數num被省略，則num和當前選中的快照的tracepoint編號相同。
　　tfind pc addr
　　選擇和特定地址addr相關的下一個快照。當然關於地址含有outside、range等各種變體。
　　tfind line [file:]n
　　選擇和特定行號關聯的下一個快照。

7.2.2 tdump

　　該命令不接受任何參數，其用於打印當前trace snapshot所收集到的所有數據。tdump通過掃瞄跟蹤點的當前收集行為，然後按照其行為打印對應每個表達式的值。

7.3 使用Trace Files

　　可以將線上收集到的數據保存到文件當中，後續可以通過target tfile進行加載，就像使用原始的trace數據一樣訪問。
　　tsave [-r] filename
　　默認的情況下該命令保存在host的文件系統中，所以GDB將會從target上面拷貝對應的數據到本地保存。如果target支持的話，可以通過』-r'參數，直接將數據保存在target的文件系統上面，這樣在收集到的數據量很大的時候會更加的高效。
　　target tfile filename
　　加載tfile數據，使後面可以感覺和target在線訪問的效果，只不過只能訪問歷史數據而不能進行任何新的trace experiment。

八、GDB對編程語言的支持

8.1 GDB對C/C++語言的支持

8.1.1 GDB對C/C++語言的支持

　　因為總所周知的原因，GCC和GDB對C/C++的支持一直很不錯，包括在gdb命令行中的表達式、操作符的使用就可見一斑了。C++要比C複雜的多，所以下面羅列了一些GDB和C++調試的一些東西。
　　C++成員函數的調用是支持的，形如:count = aml->GetOriginal(x, y);；
　　當成員函數正在被調試(作為選中的stack frame)的時候，調試的表達式具有和成員函數一樣的名字空間；
　　在某種程度上支持C++重載函數的調用，GDB會進行解析並指向正確的函數調用。但是限制是：GDB不支持用戶定義類型轉換所支撐的函數重載，以及調用不純在的構造函數，沒有對應類型實例化的模板函數，也不能處理省略的和默認的函數參數。對於數值類型轉換、類型提升都是正常支持的。函數重載默認是使能的，除非使用set overload-resolution off關閉之，然後就可以顯式調用重載版本的重載函數了：p'foo(char,int)』(『x',13)，自然GDB的補全功能不會讓你輸的很累的。
　　GDB支持C++的::名字解析操作符，正如同函數代碼中使用方式一樣。

8.1.2 針對C++的其他GDB特性

　　針對C++的重載特性，GDB可以自動補全所有的重載版本列表，方便識別和選擇；rbreak regex這種正則形式的斷點，可以在所有重載版本的函數上實現添加。
　　catch throw|rethrow|catch 可以提供C++異常處理的監測支持。
　　ptype typename 可以顯示該類型的繼承關係，同時其成員變量和成員函數也顯示的較為詳細。
　　info vtbl expr 對多態機制虛函數表的支持。

8.2 C語言預處理宏

　　宏本身是個比較麻煩的東西，因為可以某些點定義、某些點取消定義、某些點重新定義，GDB支持對含有宏的表達式進行展開並顯示展開後結果的功能。如果要讓編譯後的程序具有宏調試功能，需要額外的編譯參數-gdwarf-2和-g3(-g是不夠的)

　　上面的gdwarf-2具有gdwarf-3、gdwarf-4等版本，推薦支持的情況下使用最新版本的調試格式。
　　macro exp|expand expression
　　顯示expression中所有宏被展開後的結果。因為GDB只會進行宏展開而不會對表達式取值，所以這裡的expression不需要是合法的表達式。
　　info macro [-a|-all] [–] macro
　　顯示當前的或者全部的macro宏定義，同時顯示該宏定義所在的源代碼文件及其所在行號位置信息。其中的–表示參數列表結束，因為C中有些宏是可以使用-開頭的，這裡用於消除歧義作用。
　　info macros location
　　顯示所有在location位置有效的宏定義及他們的位置信息。(顯示結果有點多哦)
　　macro define macro replacement-list
　　macro define macro(arglist) replacement-list
　　自定義宏及其展開式，通過該命令創建的宏會在GDB求值的每個表達式中都會生效(只對GDB本身作用，與代碼中使用的宏無關)，直到其顯式使用macro undef進行刪除，同時該宏還會覆蓋程序中同名宏的展開(有點詭異)。
　　macro undef macro
　　只會刪除上面使用macro define定義的宏，而不會刪除程序代碼中定義的宏。
　　macro list
　　顯示所有通過macro define定義的宏。
　　除了通常在源代碼中執行宏定義，還可以在編譯的時候在命令行中通過『-Dname=value'的方式定義，這樣的宏也支持使用info macro命令查看，不過其行號信息顯示為0。

九 修改調試程序的執行

　　例如在程序調試的過程中發現了明顯的錯誤，想要驗證修改該簡單錯誤後程序執行會不會得到正確結果的時候，GDB的alter execution功能就會比較的好用，支持比如：修改某個寄存器、變量、內存的值，向程序發送某個信號，從不同的地方重新執行，直接跳出函數執行等操作。

9.1 給變量重新賦值

　　下面都是將變量x修改為新值，區別是使用print會同時打印這個變量的值，同時還會降其放入歷史記錄中去。在使用set的時候還需注意，因為gdb中有很多變量也是使用set設置更新的(比如width)，此時就應該使用set variable|var而不能使用set簡寫了，並且使用後者總是一個好習慣，可以防止莫名其妙的修改了環境變量值而不知。

　　GDB中對於變量類型的轉換要比C/C++語言寬鬆的多，比如可以把整數當做指針使用等，也可以直接將值放到指定內存位置上：

9.2 從不同位置恢復執行

　　默認使用continue恢復程序執行的時候，會從之前停止的位置繼續執行，而使用jump命令可以指定程序恢復執行的位置。該命令很多時候是調回已經執行的代碼，重新設置某些變量，然後再次執行下來。
　　jump|j location
　　在location的位置恢復執行，如果該處有斷點則會立即停止，而且有時候為了達到這個效果會同tbreak組合使用。jump除了修改程序計數器之外，不會修改當前stack frame、stack pointer、內存信息和其他的寄存器信息，所以如果使用jump調到別的函數位置開始執行，那麼結果將會變得十分離奇，因此如果location的地址不在當前執行函數範圍內，GDB會給出信息要求確認。
　　將$pc直接設置新值，然後運行continue也會得到相同的效果。

9.3 向程序發送信號

　　signal sig
　　恢復程序的執行並立即給予其sig信號。如果sig為0，程序將不會收到信號並恢復執行，這常常用於程序因為收到某個信號而停下來，該信號原本會傳遞給應用程序的時候，通過』signal 0』可以讓程序不收到該信號而繼續執行。注意在多線程的環境下，當恢復執行程序的時候信號會被傳送到當前選定的線程(而可能不是最後停止下來的線程)，因此如果要使用』signal 0』屏蔽信號，必須先選中對應正確的線程，否則GDB也會探測並給出確認提示。
　　此處的signal發送信號和命令kill發送信號是不同的，後者仍然會通過GDB根據handle覺得過濾處理，而signal會直接將信號傳遞給調試的應用程序。
　　queue-signal sig
　　將sig信號排隊給當前線程，並在程序恢復執行的時候立即發送，要求sig的handle必須是pass的，否則GDB會報錯。
　　sig參數還可以是0，起效果是當前線程所有排隊的信號將會被清空，程序恢復執行的時候將沒有信號發給該線程。該命令和上面signal不同的是signal會導致程序的恢復執行，再則queue-signal發送的信號必須是pass處理的信號。

9.4 函數中返回　

　　return [expr]
　　該命令可以取消一個函數的執行，如果提供了expr表達式，則該表達式的值將會作為函數的返回值。
　　當使用了該命令，GDB會放棄當前選定的stack frame及其包含在其內部的所有stack frame，使得該函數的調用者作為最內層stack frame。return命令不會恢復程序的執行，而是使得程序的狀態處於函數剛剛返回的狀態，與其不同的是finish命令，會恢覆函數的執行直到函數函數的正常範圍時停止。

9.5 調用程序的函數

　　print expr
　　對表達式expr求值，並打印顯示結果，其表達式可以包含對被調試程序中的函數的調用。
　　call expr
　　同樣對表達式expr求值，但是不顯示void返回函數的結果，如果調用的函數本身會返回值，那麼該返回的值還是會被打印，並添加到值歷史記錄中去。

十、調試目標和遠程調試

　　GDB的調試目標可以是process(進程)、exec file、core file、recording sessions，以及突破單機限制採用串口線、網絡相連的remote調試目標。調試目標可以使用target命令進行設置，target+TAB可以查看當前gdb所支持的調試目標的種類。
　　target type parameters
　　參數type常見的有exec|core，這種方式也可以使用exec-file、core-file方式指定，甚至在gdb啟動的時候就作為啟動參數附加上去。target工具最常用的，還是在於對遠程調試的支持上。遠程調試，是主要在於一些運行程序的遠程操作系統的限制(比如內核限制、內存限制)導致無法全功能運行gdb調試器的情況下尤為的有用，此外就是gdbsever和gdb規定了一套通信協議，而前者尺寸比後者要小的多，運行起來更輕便的同時，也容易在一個新平台上面進行快速的移植和支持。

10.1 連接到遠程調試目標

10.1.1 兩種遠程連接模式

　　GDB支持兩種模式的遠程調試連接：remote和extend-remote，兩者基本功能相同，但是在處理出錯、退出等細節性的方面差異很大。
　　target remote
　　當調試的程序退出或者detach後(還比如在gdb中使用kill主動殺死正在調試的進程)，target將會斷開連接，當使用了gdbserver的時候，gdbserver也會自動退出。
　　gdbserver的啟動模式有三種：PROG、–attach、–multi，前兩者是在gdbserver啟動的時候指定調試的二進製程序或者要attach的調試進程，在remote模式下只能連接這兩種模式的gdbserver。
　　不支持run命令，連接到遠程後被調試程序已經運行了，此時可以使用step、continue等各項調試指令。
　　不支持attach命令，必須在啟動的時候通過–attach參數指定。
　　target extend-remote
　　當調試的程序退出或者detach後，target任然保持著和gdbserver連接(此時只有gdb退出後，gdbserver才會reopen偵聽)，即使當前沒有調試程序在運行。使用–once參數可以讓gdbserver在第一個連接斷開後退出，否則如果需要使遠程的gdbserver退出，可以使用monitor exit命令。
　　extend-remote除了上面PROG、–attach外，還可以連接使用–multi模式啟動的gdbserver，連接後在使用命令設置遠程調試文件或者遠程attach進程。
　　支持使用run命令，啟動的程序是通過set remote exec-file的方式指定的。如果gdbserver啟動的時候指明了調試文件，那麼也不需要使用run命令了，可以類似使用step、continue等命令恢復程序執行。
　　支持attach命令。

10.1.2 指明Host和Target文件

　　具有調試符號的執行文件才可以用於調試。在遠程調試模式下，允許GDB通過建立的調試連接訪問遠程的程序文件(remote program file access)。
　　我們將運行gdb的成為Host，運行gdbserver的稱為Target，如果遠程程序文件訪問是支持的，那麼允許Target調試的程序是stripped之後的(確實調試符號)，只需要Host加載的程序是帶有調試符號的就可以了，除此之外還需要使用set sysroot的方式指明其他組件和庫的調試信息。即使此時Host和Target的程序不一致，但是也必須保證後者是前者strip得到的，否則調試將會有異樣的結果。

10.1.3 一些遠程調試相關命令

　　target remote|extended-remote serial-device
　　target remote|extended-remote [tcp:]host:port
　　host主機可以是主機名或者IP地址，如果host和target運行在同一台主機上，則host字段可以被省略(:佔位還是需要保留)。
　　target remote|extended-remote udp:host:port
　　target remote|extended-remote | command
　　在後台運行command，並通過管道與之通信，command必須是shell的命令並使用/bin/sh進行解析。

　　detach
　　當完成調試的時候，可以使用該命令進行對GDB控制的釋放，當使用remote模式連接的時候，此時GDB可以自由連接其他的target了；而如果使用extend-remote連接的時候，此時GDB仍然處於和target連接狀態。
　　disconnect
　　斷開和target的連接，此時GDB可以自由連接其他的target了。
　　monitor cmd
　　允許想遠程發送任意命令，被遠程monitor所解析。motinor exit可以讓gdbserver立即退出，通常在disconnect命令之後使用，而monitor set debug|remote-debug 0|1還可以設置後續描述到的這兩個調試參數。

　　gdb允許想遠程目標上傳、接收、刪除文件，這對於嵌入式調試(已佔用串口線)十分方便，命令如下：
　　remote put hostfile targetfile
　　remote get targetfile hostfile
　　remote delete targetfile

10.2 gdbserver程序

　　如前面所說，gdbserver運行的Target端可以是strip後沒有調試符號的運行程序，而在Host端負責符號相關的支持。OPTIONS參數選項比較重要的有–debug，可以顯示一些額外的調試信息，而–remote-debug會顯示一些遠程調試協議相關的信息，主要是gdbserver本身開始調試使用的。
　　gdbserver [OPTIONS] COMM PROG [ARGS …]
　　命令中的COMM是串口設備字段或者TCP/IP的網絡字段，ARGS是作為給PROG的運行參數使用。
　　gdbserver [OPTIONS] –attach COMM PID
　　調試一個運行著的進程的時候，不需要指明運行的二進製程序的位置。在extended-remote模式下還允許使用attach命令進行進程指定。pid可以使用pidof工具輔助查找(有多個同名運行的進程的話會一併全部返回，使用-s可以返回單個PID)。
　　gdbserver [OPTIONS] –multi COMM
　　允許gdbserver在不指定調試程序、調試進程的情況下啟動，後續通過extended-remote連接後再行設定。

十一、GDB TUI(Text User Interface)調試界面

　　就像之前說到的，Windows下面的程序員是幸福的，因為他們有著號稱最好用的IDE——Visual Studio。其實內行看門道，到這裡發現GNU GDB也是當之無愧的程序調試利器，只不過像通常Linux平台下的軟件一樣擅長功能而不善於表達，導致給外人看來一種很難用的「假象」。
　　GDB內部集成了一個TUI的用戶界面，在啟動GDB的時候可以使用gdb -tui參數的方式使能，該模式是使用的curses庫實現的一個建議UI界面。這個界面基本只是顯示的基本功能，遠遠達不到IDE效果，不過它的好處是比較的易用(因為他沒啥功能……)，只需要在command窗口進行常規的gdb調試，分割出來的其他窗口可以自動顯示源代碼、反彙編、寄存器等信息，操作要有好許多。該模式下箭頭可以滾動源代碼子窗口，而Ctrl-P、Ctrl-N組合鍵可以瀏覽以前的GDB歷史命令。
　　TUI在list代碼、當前執行位置等信息的顯示上比命令模式的GDB確實要方便很多，但是困擾我的是很多終端更新不及時顯示錯亂，需要不短的刷新界面才能正常顯示。此外，在Linux下面還有一個十分常用的gdb調試外殼，就是DDD，感興趣的也可以去瞭解一下。
