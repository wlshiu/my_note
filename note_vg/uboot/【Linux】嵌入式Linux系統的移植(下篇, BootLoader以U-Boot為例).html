【Linux】嵌入式Linux系統的移植（下篇：BootLoader，以U-Boot為例）

BootLoader

BootLoader的概念

BootLoader就是在操作系統內核運行之前運行的一段小程序。通過這段小程序，可以初始化硬件設備、建立內存空間的映射圖，從而將系統的軟硬件環境帶到一個合適的狀態，以便為最終調用操作系統內核準備好正確的環境，這就是所謂的引導加載程序（Boot Loader）。

在Flash存儲中存放文件的分佈圖如下所示：

歸根結底，BootLoader的任務是引導操作系統，所謂引導操作系統，就是啟動內核，把內核加載到內存RAM中去運行。

本文中，BootLoader的起始地址為0x20000000，內核（uImage）的起始地址為0x20008000，根文件系統（initrd.img.gz）的起始地址為0x21000000。

BootLoader的分類

通常，BootLoader是嚴重地依賴於硬件而實現的，特別是在嵌入式中。因此，在嵌入式裡建立一個通用的BootLoader幾乎是不可能的。因此，BootLoader的種類繁多，比如：

可以看出，不同的bootloader具有不同的使用範圍。

其中最令人矚目的就是有一個叫U-Boot的BootLoader，是一個比較通用的引導程序，而且同時支持X86、ARM和PowerPC等多種處理器架構。

U-Boot

U-Boot，全稱 Universal Boot Loader，是遵循GPL條款的開放源碼項目，是由德國DENX小組開發的用於多種嵌入式CPU的BootLoader程序，對於Linux的開發，德國的u-boot做出了巨大的貢獻，而且是開源的。

U-Boot具有以下特點：

  1. 開放源碼；
  2. 支持多種嵌入式操作系統內核，如Linux、NetBSD, VxWorks, QNX, RTEMS, ARTOS, LynxOS；
  3. 支持多個處理器系列，如PowerPC、ARM、x86、MIPS、XScale；
  4. 較高的可靠性和穩定性；
  5. 高度靈活的功能設置，適合U-Boot調試、操作系統不同引導要求、產品發佈等；
  6. 豐富的設備驅動源碼，如串口、以太網、SDRAM、FLASH、LCD、NVRAM、EEPROM、RTC、鍵盤等；
  7. 較為豐富的開發調試文檔與強大的網絡技術支持；

其實，把u-boot可以理解為是一個小型的操作系統。

U-Boot的常用命令

U-Boot的文件通過SD卡或者USB線可直接燒寫到Flash中，然後就可以進行U-Boot的命令的執行了。

U-Boot的基本命令

  • print：打印U-Boot中的環境變量（打印格式：變量名=變量值）
  • setenv：設置U-Boot中的環境變量，重啟後無效（設置格式：setenv 變量名 變量值；當變量值為空的時候，表示刪除該環境變量）
  • saveenv：保存U-Boot中的環境變量，重啟後有效
  • md：內存顯示，顯示特定內存地址的內容（設置格式：md 內存地址，如20008000）

U-Boot網絡層相關命令

一般而言，將文件（例如：內核）從主板傳輸到目標板採用TFTP協議，這就涉及到U-Boot的網絡層設置。而U-Boot的網絡層的設置與環境變量ipaddr有關：

setenv ipaddr 192.168.10.3
saveenv

需要注意的是，該ip地址需要和虛擬機的網卡地址在同一網段，即需要Ping通（U-Boot為了精簡，Ping協議並不完全，必須由目標板去Ping主機虛擬機）。這樣才能完成目標板與主板虛擬機的網絡層的通暢。

Ping通之後，就可以通過TFTP進行文件傳輸了。由於U-Boot內有TFTP的客戶端，因此服務器端有兩種可能：

  • 主板：下載TFTPD軟件，設置目錄並選擇網卡

  • 主板虛擬機：下載服務器端軟件，命令如下：

    sudo apt-get install tftpd-hpa //32位 sudo apt-get install tftpd openbsc-xinetd //64位 然後TFTP的目錄配置文件在/etc/inetd.conf中，比如：

    sudo vi /etc/inetd.conf ... tftp dgram udp wait noboby /usr/sbin/tcpd /usr/sbin/in.tftpd [目錄] ... sudo /etc/init.d/tftpd-hpa/restart //重啟服務 目標板進行TFTP需要知曉服務器端的ip地址、端口號、傳輸到的內存地址和需要傳輸的內容。服務器端的ip地址在環境變量serverip，端口號是TFTP規定好的（69號端口），剩下的兩個參數就是tftp命令需要指定的：

    tftp 20008000 hello.c 這段命令的意思為，從環境變量serverip的服務器端將hello.c文件通過TFTP協議傳輸到目標板內存地址為20008000的內存中。

U-Boot Flash相關命令

當調試完程序後，最終需要將其寫入NandFlash中，否則程序仍在內存中，一旦重啟就會消失。這就需要用到nand命令了。其一般格式為：

nand erase [NandFlash內部地址] [擦除大小]
nand [write/read] [內存地址] [NandFlash內部地址] [搬移大小]

通常，在使用write之前都需要先進行erase命令。例如：

nand erase 500000 1024            //擦除Flash5M地址1字節的內容
nand write 20008000 500000 1024            //將內存20008000以後1字節的內容寫入Flash5M地址
nand read 20008000 500000 1024            //將Flash5M地址以後1字節的內容讀取到內存20008000

這樣就能保證，目標板重啟之後數據也不會丟失了。

U-Boot啟動內核命令

bootm命令用於啟動一個操作系統映像。它會從內核映像文件（uImage）的頭部取得一些信息，這些信息包括：映像文件的基於的cpu架構、其操作系統類型、映像的類型、壓縮方式、映像文件在內存中的加載地址、映像文件運行的入口地址、映像文件名等。

緊接著bootm命令將映像加載到指定的地址，如果需要的話，還會解壓映像並傳遞必要有參數給內核，最後跳到入口地址進入內核。

U-Boot啟動內核，需要滿足一定的條件：

  • 啟動參數 bootargs
  • 根文件系統

啟動參數bootargs，一般包括以下三項：

  1. root=：啟動的根文件系統的位置相關信息
  2. init=：內核啟動後的第一個可執行文件，即init進程
  3. console=：內核啟動後，作為控制台的設備

根文件系統一般有兩種方式：Ramdisk、NFS。

Ramdisk：顧名思義內存磁盤，即將根文件系統放在內存中。本質上，RamDisk並非一個實際的文件系統，而是一種將實際的文件系統裝入內存的機制，並且可以作為根文件系統。實際上它使用的文件系統是ext2。

此時的配置需要指定root的位置在RAM，同時還需要指定RAM中的具體位置和大小，啟動參數為：

setenv bootargs root=/dev/ram initrd=0x21000000,8M init=/linuxrc console=ttySAC0,115200
saveenv

確定完啟動參數之後，就可以傳輸文件，並進行內核啟動了，使用bootm命令：

tftp 20008000 uImage
tftp 21000000 initrd.img.gz
bootm 20008000

NFS：顧名思義網絡文件系統。即允許網絡中的計算機之間通過TCP/IP網絡共享資源。在NFS的應用中，本地NFS的客戶端應用可以透明地讀寫位於遠端NFS服務器上的文件，就像訪問本地文件一樣。這種方式，對於調試就非常方便。

簡單一點說，就是將根文件系統放在主板的虛擬機上，目標板通過NFS來進行訪問。那麼，顯而易見，主板的虛擬機需要安裝NFS的服務器端。

sudo apt-cache search nfs-                //記不得名稱了可以用此命令搜索
sudo apt-get install nfs-kernel-server

安裝完成後，需要進行NFS的配置，配置文件在/etc/exports中：

sudo vi /etc/exports
...
[目錄] *(rw,sync,no_subtree_check)
...
sudo /etc/init.d/nfs-kernel-server restart        //重啟服務

需要注意的是，根文件系統的後綴為.gz，是一個壓縮包。當通過TFTP傳輸給目標板之後，目標板會自動對其進行解壓操作，形成相應的文件夾。然而，如果採用NFS的方式，服務器端需要自身解壓完，留給目標板進行訪問：

gunzip initrd.img.gz            //解壓為initrd.img，相當於一個ios文件
mount -t ext2 init.img [掛載目錄]                 //掛載initrd.img

服務器端配置完成後，目標板的配置需要指定root的位置在NFS，同時還需要指定NFS中的具體ip、端口和目錄絕對地址，端口號是NFS規定好的。

除此之外，U-Boot一旦啟動內核之後，它就喪失了對目標板的主動權，此時它的環境變量ipaddr也喪失作用了。這個時候，目標板就是屬於無ip的狀態，自然NFS就無法連接上了。因此，還需要設置ip參數。啟動參數為：

setenv bootargs root=/dev/nfs nfsroot=192.168.10.110:/home/rocky/work/rootfs ip=192.168.10.120 init=/linuxrc console=ttySAC0,115200
saveenv

確定完啟動參數之後，就可以傳輸文件，並進行內核啟動了，使用bootm命令：

tftp 20008000 uImage
bootm 20008000

採用NFS方式的好處就是，可以在主板虛擬機上寫一個程序，並運用交叉編譯器進行編譯，編譯完成後，自動就會同步到目標板上。這個時候，就可以直接在目標板上運行編譯完的程序。即主板虛擬機編譯，目標板運行！

U-Boot自動啟動內核命令

可以看出，上面的兩種啟動內核的方式都不能做到一上電就自動啟動內核的作用。其實這都和U-Boot的環境變量有關：bootdelay、bootcmd。前者為啟動後倒計時，後者為倒計時後自動執行的命令。

就可以將內核、根文件系統都nand到Flash中，倒計時後從Flash中讀取並啟動內核：

setenv bootdelay 2
setenv bootcmd nand read 20008000 100000 200000;nand read 21000000 5600000 400000;bootm 20008000
